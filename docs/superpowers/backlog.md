- Тема готова к старту → запускаем `/brainstorming` (для новых фич) или сразу

- Перед стартом проверить блок «Предпосылки» темы — если не закрыты, сначала
  закрыть или осознанно строить fallback.

## 1. Светофор авто-проверок и контекст карточки заявки

**Статус: спека и план готовы (15.07.2026)** →
`specs/2026-07-14-verdict-traffic-light-design.md`,
`plans/2026-07-15-06-verdict-traffic-light.md`; выполнять после плана 03.
Решения: 5 проверок в v1 (финплан и «заранее согласовано» — после доставки
финмодели в DWH), сводка внутри реестра, гибридные источники срезов
(API 1С / DWH / fixture per slice); боевые адаптеры срезов — в план DWH.

**Что это:** перенос «мозгов» старого приложения — автоматическая оценка
заявки перед согласованием (🟢/🟡/🔴/⚫) + контекст для решения за 2–3 секунды.

**Состав (из ТЗ и старого кода):**
- 7 проверок: деньги на счёте, остаток фонда, соответствие финплану,
  документ-основание, договор, история контрагента, предварительное
  согласование. Совокупный вердикт = худшая из обязательных проверок.
- Секции карточки: ликвидность, фонд (план/факт/нагрузка), финплан,
  история контрагента (платежи, дебиторка/кредиторка), договор/основание,
  вложения, связанные заявки (±30 дней).
- Сводный экран: остатки по р/с в валютах с проекцией «после отмеченных»,
  карточки фондов (клик = фильтр), группировка заявок по фондам, метрики.

**Первоисточники (репозиторий `fin`):**
- `ТЗ_Карточка_заявки.docx` — правила светофора (пороги: фонд в минусе ≤20% →
  warn, превышение плана ≤15% → warn, «постоянный» контрагент ≥3 операций).
- `composables/useVerdict.ts`, `types/index.ts` (Nuxt) и
  `finflow-app/frontend/src/reports/requestVerdict.js` — рабочая реализация.
- Макеты: `Макет карточки заявки [cjlo4r].html`, `Сводный экран v2 c p c [0kMklY].html`.

**Предпосылки:**
- Срезы в DWH: остатки на счетах, фонды (план/баланс/движения),
  финмодель/бюджет, взаиморасчёты с контрагентами, договоры, вложения.
  Заказать вьюхи владельцу DWH — тем же путём, что и заявки (план 04).

**Зависимости:** после плана 04 (реальный DWH), поверх ядра из плана 03.

## 2. План 04 — реальное подключение к DWH (mssql) + cron

**Статус: план готов (15.07.2026)** → `plans/2026-07-15-08-dwh-mssql-cron.md`;
выполнять после планов 03 и 06. Включает боевые адаптеры срезов светофора
(DWH + API 1С) и контракт вьюх для заказа владельцу DWH
(`docs/contracts/dwh-views.md`). Блокирующие предпосылки: вьюхи по контракту,
SQL-логин, адрес DEEONE, SSH-ключ на VPS.

**Что это:** не брейншторм, а план по уже утверждённой спеке (§7). Замена
fixture-шлюза на mssql-адаптер `DwhGateway` (SQL Server DEEONE через
SSH-туннель host02 → ai01) + расписание синка на сервере
(9:00–12:00 МСК каждые 15 мин, затем раз в час до 19:00).

**Предпосылки:**
- Контракт вьюх DEEONE: заявки, списания («Расход ДС»), привязка
  списание→заявка. Заказать владельцу DWH. Заодно сверить кодировку
  статусов согласования (открытый вопрос §11.8 спеки).
- Доступ по SSH с сервера приложения (ключ, цепочка — образец:
  `fin/finflow-app/dwh.py`).
- npm-зависимость `mssql` — одобрена Минасом в спеке.

**Зависимости:** контур доставки (план 02) — cron живёт на сервере.

## 3. План 05 — отправка платёжек поставщикам (Яндекс Мессенджер)

**Что это:** план по утверждённой спеке (§8): таблицы `PaymentOrderDispatch`
и `CashFlowItemSetting`, экран `app/dispatch/` (очередь на подтверждение +
журнал), настройки статей ДДС «оплата за товар», интеграция с ботом ЯМ.
Старт возможен в ручном режиме (бухгалтер прикрепляет файл и указывает чат)
до закрытия предпосылок.

**Предпосылки (§11 спеки, закрываются вне кода):**
- Процесс выкладки файлов платёжек в папку на сервере: кто, куда, формат
  имён (определяет правило сопоставления файл↔списание).
- Бот Яндекс Мессенджера (Яндекс 360): создать, получить токен, добавить
  в чаты поставщиков.
- Поле «ссылка на чат» в карточке контрагента 1С → доставка в DWH.

**Зависимости:** план 04 (реальные списания из DWH) — иначе очередь пуста.

## 4. Авторизация и роли в приложении

**Статус: спека и план готовы (15.07.2026)** →
`specs/2026-07-15-auth-roles-design.md`,
`plans/2026-07-15-07-auth-roles.md`; выполнять после планов 03 и 06.
Решения: своя cookie-сессионная авторизация без новых зависимостей
(scrypt из node:crypto, сессии в PostgreSQL), 3 роли, basic auth убирается
на обоих контурах после релиза.

**Что это:** замена nginx basic auth на настоящую авторизацию. Ролевая
модель уже зафиксирована концептуально в спеке заявок (§3):

| Роль | Права в модуле заявок |
|---|---|
| Собственник / руководитель | просмотр, согласование/отклонение |
| Бухгалтер | комментарии к красным, подтверждение отправки платёжек, статьи ДДС |
| Читатель | только просмотр |

**Заметки:** в спеке окружения (`2026-07-13-team-environment-design.md`)
авторизация помечена как «фича на потом». После её появления — привязать
`author` в комментариях и `confirmedBy` в отправках к реальным пользователям
(сейчас это свободные строки).

**Зависимости:** самостоятельная тема, отдельный брейншторм.
