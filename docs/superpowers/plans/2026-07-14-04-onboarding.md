# План 4: Onboarding специалистов (GitHub-аккаунт обязателен)

**Goal:** Специалист за одну сессию `/onboarding` получает рабочее окружение:
локальный dev, доставка в dev-окружение через /ship, опер-режим. Плюс актуализация actions.

**Решение по доступу к GitHub (проверено по flow):** специалисту от GitHub нужен только
push в `feature/*`/`develop` и чтение статуса CI. Технически это возможно и без аккаунта
(deploy key + публичный API), но такая схема ломается при возврате репозитория в приватный
режим и не даёт аудита. Поэтому: **GitHub-аккаунт обязателен** — если его нет, он создаётся
прямо в ходе onboarding; разработчик добавляет специалиста коллаборатором с правом write
(бесплатно, в т.ч. для приватных репо). PR в production создаёт ТОЛЬКО разработчик —
`/request-prod` для специалистов закрыт guard'ом.

## Задачи

1. **Actions v4 → v5** в ci.yml / deploy-dev.yml / deploy-prod.yml (снимает предупреждение
   о Node 20). Никакого workflow request-prod.yml — релиз запускает только разработчик.
2. **/request-prod — только для разработчика:** первым шагом команда проверяет
   `gh api user -q .login` == `webkoth`; иначе — сообщение «релиз в production запускает
   только разработчик, сообщи ему о готовности фич» и стоп.
3. **/onboarding — боевая команда** (пошагово для Claude на машине специалиста):
   имя/email → определение ОС → установка git, Node 26, PostgreSQL 18, gh
   (winget / brew / apt+NodeSource+PGDG) → GitHub-аккаунт (создать при отсутствии,
   включить 2FA) → `gh auth login` → передать логин разработчику и дождаться права push →
   `git config` → клон → локальная БД (Windows: роль finflow/finflow; macOS/Linux:
   пользователь ОС) → `.env` → `npm install` → миграции + seed → ключ машины
   `~/.ssh/finflow_ed25519` + блок `finflow-ops` в `~/.ssh/config` (pubkey — разработчику) →
   Playwright chromium → проверка `npm run dev` → doctor.
4. **scripts/onboarding-check.mjs** — кросс-платформенный doctor без зависимостей:
   PASS/FAIL/WARN по пунктам: node>=26, git+идентичность, psql, .env, миграции применены
   (`prisma migrate status`), gh-авторизация и право push, ключ машины, ssh до ops (WARN,
   если ключ ещё не добавлен), доступ к dev-окружению (WARN — пароль basic auth).
5. **Runbook разработчика** `docs/superpowers/runbooks/add-specialist.md`: выдача
   (collaborator push, ops-ключ, пароль basic auth, RO-строка) и отзыв доступов.
6. **README**: пометку «/onboarding активируется этапом 5» заменить на «активна».

## Приёмка
- Прогоны CI на v5 зелёные, предупреждение Node 20 исчезло.
- `node scripts/onboarding-check.mjs` на машине разработчика: все проверки корректно
  срабатывают (PASS там, где настроено).
- Guard /request-prod: под разработчиком пропускает, логика ветки «не разработчик» на месте.
