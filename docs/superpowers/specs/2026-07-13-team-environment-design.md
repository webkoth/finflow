# Спецификация: рабочее окружение команды finflow

Дата: 2026-07-13
Статус: утверждена (гриллинг-сессия), ожидает план внедрения
Автор решения: Минас (разработчик/ревьюер) + Claude

## 1. Цель

Стандартизированное окружение Claude Code для 2–5 доменных специалистов (не разработчиков),
которые совместно развивают одно приложение **finflow** и работают в двух режимах:

- **Режим разработки** — добавление функциональности своего домена в приложение
  по управляемому процессу (идея → спека → план → код → тесты → доставка) без погружения в код.
- **Операционный режим** — ежедневная доменная работа: отчёты и выгрузки по реальным данным,
  контроль состояния приложения (статус, логи, перезапуск dev-окружения).

Правила и рельсы существуют, чтобы соблюдались конвенции и архитектура не деградировала
при параллельной работе нескольких не-разработчиков.

**Терминология:** в устном общении со специалистами dev-окружение называется
«песочницей» — так им понятнее. Де-факто и де-юре это одно и то же: ветка `develop`,
dev-контур, dev.<домен>. Во всех технических артефактах — только dev/develop.

## 2. Роли

| Роль | Кто | Права |
|---|---|---|
| Разработчик | Минас | Ревью PR в main, правка темы/конвенций, prod-операции, донастройка окружений |
| Специалист | 2–5 доменных экспертов | Разработка в feature/*, merge в develop, опер-режим (RO-данные, dev-эксплуатация) |

## 3. Стек приложения (зафиксирован, отступления запрещены)

- **Приложение**: Next.js 16 (App Router) + React 19 + TypeScript. Только Next.js — другие
  фреймворки/стеки не используются ни при каких условиях.
- **UI**: Tailwind CSS 4 + shadcn/ui (вариант на Base UI) — см. раздел 10.
- **БД**: PostgreSQL + Prisma (schema.prisma — единственный источник истины схемы).
- **Тесты**: Vitest (unit, только доменная логика) + Playwright (e2e-смоук).
- **Рантайм**: Node.js 26 (зафиксировать в `package.json engines` и `.nvmrc`), пакетный менеджер npm.

## 4. Инфраструктура

- Один VPS (без Docker; при росте — переезд dev/prod на отдельные серверы).
- Процессы: pm2 — `finflow-dev` и `finflow-prod`, разные порты.
- nginx: два vhost-поддомена (dev.<домен> и <домен>), оба закрыты **basic auth**
  (общий логин/пароль). Настоящая авторизация в приложении — отдельная будущая фича.
- PostgreSQL на том же VPS: базы `finflow_dev` и `finflow_prod`.
- Роли Postgres:
  - `finflow_app_dev` (rw, только dev-БД) — для dev-приложения;
  - `finflow_app_prod` (rw, только prod-БД) — для prod-приложения;
  - `finflow_ro` (read-only на prod-БД) — для опер-режима специалистов.
- SSH-пользователи:
  - `deploy` — используется только GitHub Actions (ключ в GitHub Secrets);
  - `ops` — для специалистов: логи и статус обоих контуров, перезапуск **только** dev,
    запуск reset-скрипта dev-БД. Прод перезапускает только разработчик.

## 5. Git-flow и доставка

Репозиторий: GitHub — https://github.com/webkoth/finflow (ветки `main` и `develop`
запушены). Пока репозиторий **публичный** — на период обкатки процесса; после
обкатки переведём в приватный. Секреты в репо не попадают в любом случае
(см. раздел 7), но до перевода в приватный это правило особенно критично.

```
feature/<слаг> ──merge──▶ develop ──CI──▶ деплой dev.<домен>
                             │
                             └──PR + ревью разработчика──▶ main ──CI──▶ деплой <домен>
```

- `feature/*` — рабочие ветки специалистов, создаются автоматически в начале фичи.
- `develop` — интеграционный транк. Merge из feature/* без ревью. Push → автодеплой в dev-окружение.
- `main` — production. **Branch protection на GitHub**: только через PR develop→main
  с апрувом разработчика; прямые пуши запрещены всем. Это единственный жёсткий замок
  процесса, не зависящий от Claude. Релизы идут пачками (что накопилось в dev).
- Побочное следствие модели: сырая фича в develop блокирует релиз пачки —
  лечится мелкой нарезкой фич (за этим следит brainstorming-этап декомпозиции).

### CI/CD (GitHub Actions)

| Триггер | Шаги |
|---|---|
| push в любую ветку | lint → typecheck → build → unit-тесты |
| push в develop | + e2e-смоук → деплой по SSH (build на раннере, rsync) → `prisma migrate deploy` на dev-БД → pm2 reload → HTTP-проверка живости |
| merge в main | те же шаги против prod |

- Миграции применяются **автоматически** при деплое обоих контуров; для prod SQL миграций
  уже виден разработчику в диффе PR.
- Секреты CI (SSH-ключ deploy, хосты, строки БД) — только в GitHub Secrets.
  У специалистов ключей от сервера уровня deploy нет.

## 6. База данных: правила

- Схему меняем только через `schema.prisma` + `prisma migrate dev` (создаёт файл миграции).
- Применённые миграции никогда не редактируются — только новая миграция.
- В репо живёт seed-скрипт (`prisma/seed.ts`) с реалистичными демо-данными.
- Данные dev — расходные: команда `/reset-dev` пересоздаёт dev-БД
  (миграции + seed) с подтверждением и уведомлением команды.
- Деструктивные миграции (drop колонок/таблиц с данными) — Claude обязан явно
  предупредить специалиста и пометить это в описании PR.

## 7. Локальная разработка и onboarding

- Специалист работает локально: dev-сервер Next.js + **локальный** PostgreSQL.
- Машины: преимущественно Windows, есть macOS и Linux → **вся автоматизация
  (скрипты, хуки, onboarding) пишется на Node.js**, не на bash.
- GitHub-аккаунт обязателен каждому специалисту (создаётся в ходе onboarding при
  отсутствии; разработчик добавляет collaborator с write). PR в production при этом
  создаёт только разработчик — /request-prod закрыт guard-проверкой логина.
- Onboarding-команда `/onboarding` определяет ОС и выполняет: установку
  Node 26 и PostgreSQL (winget / Homebrew / apt), создание локальной БД,
  генерацию `.env` из `.env.example`, `npm install`, миграции + seed,
  установку браузеров Playwright, проверочный запуск `npm run dev`.
- Разработчик лично выдаёт специалисту: write-доступ к GitHub-репо (collaborator;
  пока репо публичное, читать может любой — см. раздел 5), строку `DATABASE_URL_PROD_RO`,
  SSH-ключ пользователя `ops`, пароль basic auth.
- Секреты не коммитятся: локально `.env` (gitignore), на сервере — env-файлы,
  управляемые только разработчиком, в CI — GitHub Secrets.

## 8. Процесс разработки (мягкие рельсы)

- Любая новая функциональность начинается со скилла **brainstorming** и его цепочки:
  спека → `writing-plans` → `executing-plans` (внутри — TDD-скилл, ограниченный
  тестовой стратегией из раздела 9). HARD-GATE скилла запрещает писать код
  до утверждённого дизайна.
- Скиллы процесса — **копиями в репо** `.claude/skills/` (brainstorming, writing-plans,
  executing-plans, test-driven-development, systematic-debugging), чтобы не зависеть
  от личных установок на машинах. Обновляет их разработчик коммитом.
- Конвенции (структура каталогов, стек, правила БД и UI, правило зависимостей)
  описаны в `CLAUDE.md` репозитория. Жёстких хуков-блокировок процесса нет —
  принуждение мягкое, страхуется ревью PR в main и branch protection.
- Правило зависимостей: **любая новая npm-зависимость — только с одобрения разработчика**;
  контроль — дифф `package.json` на ревью.

## 9. Тестовая стратегия (гибрид по слоям)

- **Unit (Vitest, через TDD-скилл)** — только для чистой доменной логики в `lib/domain/`
  (расчёты, деньги, агрегации, преобразования). Это машинная спецификация
  финансовой математики.
- **E2e-смоук (Playwright)** — ключевые пользовательские сценарии
  («страница открылась, запись создалась/сохранилась/отобразилась»).
  Claude пишет и чинит их при каждой фиче.
- **Unit-тесты на UI-компоненты и страницы — запрещены** (правило CLAUDE.md):
  тавтологичны, создают ложные красные CI у чужих специалистов.
- CI-гейты: lint+typecheck+build+unit — на каждый пуш; e2e — перед деплоем
  dev и prod.

## 10. UI-конвенция: только shadcn в рамках темы

1. Компоненты — только shadcn/ui: `npx shadcn add <component>` либо композиции
   из уже установленных shadcn-примитивов. Другие UI-библиотеки запрещены.
2. Стилизация — только Tailwind-классы и токены темы (CSS-переменные `globals.css`).
   Без хардкод-цветов, инлайн-стилей, своих шрифтов. Тему меняет только разработчик.
3. Иконки — только `lucide-react`.
4. Графики — только shadcn charts (обёртка Recharts).
5. Исключения: если нужного компонента нет в shadcn — Claude останавливается
   и спрашивает разработчика; одобренное исключение фиксируется в списке
   разрешённых отступлений в `CLAUDE.md`.

## 11. Команды (в `.claude/commands/` репозитория)

| Команда | Что делает |
|---|---|
| `/ship` | Локальные проверки (lint, typecheck, build, тесты) → коммит → актуализация и merge в `develop` → push → контроль прогона Actions → ссылка на dev-окружение |
| `/request-prod` | Создаёт PR develop→main через `gh`, собирает описание изменений с момента прошлого релиза, назначает разработчика ревьюером |
| `/status` | Состояние обоих контуров: HTTP-проверка, pm2-статус через ops-SSH, последние деплои Actions |
| `/logs [контур]` | Хвост логов контура через ops-SSH |
| `/reset-dev` | С подтверждением: пересоздание dev-БД (миграции + seed) через ops-SSH |
| `/onboarding` | Первичная настройка машины специалиста (раздел 7) |

Доставка намеренно детерминирована командами; разработка (раздел 8) — мягкая.

## 12. Операционный режим

- **Данные**: отчёты/выгрузки Claude выполняет SQL-запросами под ролью `finflow_ro`
  (строка подключения в локальном `.env`). Запись в prod-данные технически невозможна.
- **Эксплуатация**: `/status`, `/logs` — оба контура; перезапуск — только dev
  (ограничение прав пользователя `ops` на сервере, не договорённость).
- Профильные потребности (интеграции, форматы отчётов) оформляются доменными
  скиллами в репо по мере появления.

## 13. Дистрибуция и кастомизация окружения

- Всё окружение — в репозитории finflow: `CLAUDE.md`, `.claude/skills/`
  (процессные + профильные доменные), `.claude/commands/`, `.claude/settings.json`
  (permissions-allowlist для типовых операций, чтобы специалистов не заваливало
  запросами разрешений).
- Обновление у всех = `git pull`. Донастройка специалисту под профиль =
  обычный коммит разработчика (например, новый доменный скилл).
- Личный слой (не версионируется, не забота разработчика): `.claude/settings.local.json`
  и `~/.claude` специалиста.

## 14. Вне объёма (сознательно)

- Авторизация/роли внутри приложения finflow (закрыто basic auth; фича на потом
  через обычный brainstorming-цикл).
- Второй сервер, preview-деплои на ветку, plugin marketplace — при росте команды.
- Доменная функциональность finflow (первые фичи пойдут уже по этому процессу).
- Жёсткие хуки на процесс разработки (решение: мягкие правила + ревью).
