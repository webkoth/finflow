# Спека: отчёт «ИУ WB план-факт»

Дата: 2026-07-15
Статус: дизайн утверждён (брейншторминг с Минасом), готово к плану реализации.
Источники: `fin/finflow/02-iu-report.md` (мигрированные метрики и решения по
расчётам), `fin/finflow/01-data-sources.md` (источники DWH), рабочая
реализация `fin/finflow-app/metrics.py` + `IuWbReport.jsx`, факт-кэш
`fin/finflow-app/data/ИУ Маркетплейсы/fact_cache.json`.

## 1. Контекст и проблема

Дашборд план-факта по WB (аккаунты 1 = ООО ТОРИ БРЕНДС, 2 = ИП Бобровская):
выполнение целей контракта по выкупам и обязательства по рекламе. Живёт в
старом приложении (репо `fin`, Flask + React) с ручными процессами
(`repull_days.py` c VPN+kinit, Excel в папке по mtime). Переносим в finflow
на его архитектуру: факт — синк в PostgreSQL, план — загрузка Excel в UI,
графики — shadcn charts.

## 2. Границы

### Входит

1. **Синк факта из DWH**: 5 дневных метрик по 2 аккаунтам — `buyout_rub`,
   `orders_rub`, `adv_rub`, `buyout_qty`, `orders_qty`.
2. **Загрузка плана**: xlsx (лист «2026 ПЛАН») через UI, парсинг в Postgres,
   история загрузок.
3. **Экран отчёта**: KPI-карточки, 4 графика, таблица периодов,
   переключатель «весь год / П1 / П2».
4. **Настройки отчёта**: параметры контракта (баннер, допсоглашение,
   % обязательства, цели полугодий) + загрузка плана.

### Не входит (осознанно)

- Excel-overlay факта (наложение Excel поверх БД) — отменён: **факт только
  из БД**, «куб = истина» (решение зафиксировано ранее).
- Показы (views) — в БД их нет и в UI не используются (подтверждено
  23.06.2026), не мигрируются.
- OZON — отчёт ИУ только по WB, как в оригинале.
- Точечный добор дней вручную (`repull_days.py`) — заменяется синком
  со скользящим окном.

## 3. Ключевые решения

| Тема | Решение |
|---|---|
| Уровень переноса | На архитектуру finflow: UI читает только Postgres, данные приносит синк |
| Хранение факта | Дневные агрегаты `IuFactDaily` (метрика × аккаунт × день) — порт `fact_cache.json` |
| SQL к DWH | **Прямые запросы, портированные из `metrics.py`** (дедупы выверены до рубля) через mssql-пул плана 8; новые вьюхи не заказываются |
| План | Загрузка xlsx в UI, парсинг листа «2026 ПЛАН», история загрузок; активна последняя |
| Константы контракта | Настройки в БД с периодами действия (баннер, допсоглашение, %, цели) — правятся без релиза |
| Синк отчётов | Отдельный scope `reports`: ночной cron + кнопка «Обновить» (запросы тяжелее заявочных, в окно выписок не гоняем) |
| Новая зависимость | **`exceljs`** (чтение xlsx) — одобрена Минасом в этом дизайне |
| Заказы | Из базовой `wb_orders` srid-дедупом (вьюха `v_wb_orders_complete` теряет март/февраль) — до починки BD-командой |

## 4. Архитектура

Инфраструктура переиспользуется из существующих планов:

- **Канал к DWH** — mssql-пул и SSH-туннель плана 8 (`lib/integrations/dwh-mssql.ts`,
  `getMssqlPool`/`queryView`-паттерн). Адаптер ИУ добавляет свои запросы
  к `marketplaces` (`WBALL.*`) — те же env, тот же туннель.
- **Синк** — та же точка `POST /api/jobs/sync` с параметром `scope=reports`
  (без scope — существующий синк заявок/срезов). Журнал — `SyncRun`
  с новым полем `scope`. Ночной cron — строка в `ops/crontab.finflow`.
- **Права** — просмотр отчёта: все роли; настройки — новое действие
  `manage_report_settings` в матрице (`owner`).
- **Fixture-режим** — снапшот реального `fact_cache.json` → fixture-модуль:
  демо-отчёт показывает настоящие исторические цифры без DWH.

Поток: DWH (5 запросов) → `IuFactDaily`; xlsx → парсер → `IuPlanWeek`;
экран = чистый домен `lib/domain/iu.ts` от (факт + план + настройки).

## 5. Модель данных (Prisma)

- **`IuFactDaily`** — `metric` (enum: `buyout_rub | orders_rub | adv_rub |
  buyout_qty | orders_qty`), `accountId Int` (1 = ТОРИ, 2 = Бобровская),
  `date @db.Date`, `value Decimal(18,2)`, `syncedAt`.
  `@@unique([metric, accountId, date])`.
- **`IuPlanDay`** — план **дневной** (уточнено по факту: лист «2026 ПЛАН» —
  строки по датам): `date @db.Date @unique`, `planBuyout`, `planWb`,
  `planOrders`, `planAdv` (Decimal), `uploadId` (FK). Строки активной
  (последней) загрузки; недельная таблица отчёта агрегируется из дней.
- **`IuPlanUpload`** — `fileName`, `rowsParsed`, `uploadedById` (FK User) +
  снапшот имени, `createdAt`.
- **`IuReportSetting`** — `key @unique`, `value Decimal(18,2)`,
  `dateFrom @db.Date?`, `dateTo @db.Date?` (период действия — для баннера).
  Сид: `banner_total` (19 000 000, 25.05–07.06.2026), `adv_extra_agreement`
  (165 172 327), `adv_percent` (7), `target_buyout_h1`, `target_buyout_h2`,
  `plan_buyout_daily_ref` (справочный темп ~8,45 млн).
- **`SyncRun`** — новое поле `scope` (`requests | reports`, default
  `requests`); индикаторы свежести каждого экрана читают свой scope.

## 6. Доменная логика (`lib/domain/iu.ts`)

Чистые функции, unit-тесты на контрольных цифрах из доки миграции:

- **Баннер**: дневная реклама = `adv_rub` + равномерная доля баннера в его
  период действия (`banner_total / кол-во дней периода`).
- **KPI «Выкупы вчера»**: план = требуемый темп
  `need_daily = (цель полугодия − факт по вчера) / остаток дней полугодия`;
  рядом справочный чип `plan_buyout_daily_ref`.
- **Факт накопительно и % выполнения** — по факту на вчера
  (`fact_to_date / target`), не по последней неделе.
- **Прогноз трат рекламы** = потрачено + ДРР последних 7 завершённых дней ×
  остаток выкупов до цели.
- **Два ДРР, подписаны явно**: график «Заказы·ДРР» — траты/заказы;
  блок обязательства — траты/выкупы.
- **Обязательство по рекламе** = `adv_percent`% плановой выручки +
  `adv_extra_agreement` (для П1; П2 — без допсоглашения, с пометкой в UI).
- **Периоды**: весь год / П1 / П2 — фильтр окна расчёта; недельная
  агрегация — ISO-недели (общий хелпер ГГНН с отчётом БДР).

## 7. Экраны

### 7.1 `app/reports/iu-wb` — отчёт

- Переключатель периода (весь год / П1 / П2; у П2 пометка про отсутствие
  допсоглашения).
- KPI-карточки: выкупы вчера (факт против требуемого темпа), факт
  накопительно + % выполнения, реклама накопительно против обязательства,
  прогноз трат.
- Четыре графика (shadcn charts): недельный план-факт выкупов,
  накопительный, реклама (факт + прогноз против обязательства),
  «Заказы·ДРР».
- Таблица периодов по неделям (план/факт/отклонение).
- Строка свежести из `SyncRun` (scope reports) + кнопка «Обновить».
- Данные = сумма аккаунтов 1 + 2 (как в оригинале).

### 7.2 `app/reports/iu-wb/settings` — настройки (owner)

- Загрузка xlsx плана: аплоад → парсинг листа «2026 ПЛАН» → отчёт
  «распарсено N строк» → история загрузок.
- Таблица параметров контракта (`IuReportSetting`) с периодами действия.
- Право `manage_report_settings` (матрица прав, owner); страница отдаёт
  404 остальным.

## 8. Синк, ошибки, деградация

- `POST /api/jobs/sync?scope=reports` (тот же cron-секрет / сессия для
  кнопки). Cron — раз в день ночью (03:00 МСК, будни и выходные — данные
  дотекают ежедневно).
- Каждая метрика — независимый шаг (ошибка одной не валит остальные),
  результат по шагам — в `SyncRun` (по образцу `slices`).
- **Скользящее окно перезаписи — 35 дней** (корректировки дотекают задним
  числом); полный пересбор — параметр `full=1`.
- Гоча дат DWH (DATEFORMAT dmy) не актуальна: только параметризованные
  запросы (паттерн плана 8).
- DWH недоступен → отчёт работает на последних данных, строка свежести
  показывает ошибку (паттерн заявок).
- Пустой план (не загружен) → отчёт показывает факт, план-элементы —
  «план не загружен».

## 9. Тестирование

- **Unit (`lib/domain/iu.ts`)**: need_daily (граница — последний день
  полугодия), прогноз трат, оба ДРР, обязательство (контрольное ≈272,2 млн
  из доки), баннер (внутри/вне периода, граничные дни), % выполнения.
- **Unit (парсер плана)**: чтение эталонного xlsx-фрагмента из fixtures
  (коды недель, метрики, пустые ячейки).
- **Интеграционные**: синк метрик на замоканном адаптере (upsert,
  идемпотентность, окно 35 дней, независимость шагов).
- **E2e**: отчёт на fixtures — KPI и таблица видны, переключение полугодия
  меняет цифры, «Обновить» работает; настройки — загрузка плана (эталонный
  xlsx из fixtures), смена параметра меняет KPI; 404 для не-owner.

## 10. Этапность реализации (детали — в плане)

1. Prisma + домен `iu.ts` + fixture-факт + экран отчёта (без плана).
2. Загрузка плана (exceljs) + настройки параметров + план в KPI/графики.
3. Боевой адаптер синка (5 запросов через пул плана 8) + cron-строка +
   сверка с контрольными цифрами старого отчёта.

## 11. Предпосылки и открытые вопросы

1. **SQL-логин с правами на `marketplaces`** — та же предпосылка, что у
   плана 8 (общий запрос BD-команде).
2. **Сверка после порта**: перед включением боевого синка сверить
   недельные суммы с действующим отчётом старого приложения (допуск 0 —
   запросы идентичны).
3. **Вьюха `v_wb_orders_complete`** — заказы временно из `wb_orders`;
   после починки BD-командой перевести на куб (кандидат в бэклог).

## 12. Пополнение бэклога

- Параметры контракта 2027 (цели П2, новое допсоглашение) — вводятся через
  настройки без релиза, но напоминание о ревизии перед 2027 годом.
