# Спека: раздел «Справочники» — статьи ДДС, статьи БДР, банковские счета

Дата: 2026-07-16
Статус: дизайн (брейншторминг с Антониной), готов к плану реализации.
Источник требований: диалог со специалистом.

## 1. Контекст и проблема

В приложении finflow появляется потребность в управляемых эталонных данных
(«справочниках»): финансовые статьи и банковские счета, на которые в будущем
будут ссылаться транзакции и модуль «Заявки на оплату». Сейчас, например,
категория транзакции — свободный текст (`Transaction.category: String`), что
не даёт единого списка и контроля.

Раздел «Справочники» — контейнер, который со временем вместит несколько
справочников. В первой итерации в нём три:

1. **Статьи ДДС** (движение денежных средств).
2. **Статьи БДР** (бюджет доходов и расходов).
3. **Банковские счета**.

### Источник данных (важное решение)

Целевой источник истины для этих справочников — 1С. Но **на текущий момент
доступа к данным 1С у finflow нет** (см. §8): нет ни подключения, ни
интеграционного слоя, а согласованный канал чтения через DWH/DEEONE ещё не
построен. Поэтому в этой итерации:

- **Владелец данных — finflow.** Справочники ведутся внутри приложения (CRUD),
  наполняются **seed/тестовыми данными** (`prisma/seed.ts`).
- **Синхронизация с 1С отложена** и вынесена в отдельную будущую фичу.
  Черновик запроса к программистам 1С сохранён в §8 как приложение.
- Модель данных спроектирована так, чтобы будущий синк лёг поверх без
  переделки (стабильные идентификаторы, мягкое удаление вместо физического).

## 2. Границы

### Входит

1. Раздел `app/reference/` с витриной и тремя справочниками.
2. Справочники статей ДДС и БДР: **древовидные** (произвольная глубина),
   поля — наименование, код, тип, признак «группа/папка», описание.
3. Справочник банковских счетов: плоский список со своими полями.
4. CRUD в каждом справочнике: создать, изменить, **архивировать / вернуть из
   архива** (мягкое удаление, физического нет).
5. Seed с тестовыми данными по всем трём справочникам.

### Не входит (осознанно, следующие итерации)

- **Синхронизация/импорт из 1С** — отдельная фича, зависит от доступа (§8).
- **Привязка статей/счетов к транзакциям и заявкам.** `Transaction.category`
  остаётся текстом; связывание сущностей — отдельная фича.
- Другие справочники: валюты, контрагенты, компании группы, договоры.
- Drag&drop-переупорядочивание дерева (перенос узла — только сменой родителя
  в форме).
- Роли и права. Как и во всём приложении — доступ закрыт nginx basic auth,
  внутренней авторизации пока нет.

## 3. Архитектура и раскладка по каталогам

Подход: единая модель `Article` для статей ДДС и БДР (они идентичны по
структуре, различаются дискриминатором `kind`), отдельная модель `BankAccount`.
Код разложен по образцу `app/transactions/`: каждый справочник — своя
фича-папка со своими `page.tsx` и `actions.ts`; общее у ДДС и БДР вынесено в
`components/` как переиспользуемая композиция.

```
app/
  reference/
    page.tsx                     # витрина раздела: 3 карточки-ссылки
    cashflow-items/
      page.tsx                   # server component: Article kind=CASHFLOW → <ArticleDictionary>
      actions.ts                 # server actions create/update/archive, kind=CASHFLOW
    pnl-items/
      page.tsx                   # то же, kind=PNL
      actions.ts
    bank-accounts/
      page.tsx                   # server component: список счетов
      actions.ts                 # server actions счёта
      bank-account-form.tsx      # client-форма счёта (специфична разделу)

components/
  reference/
    article-dictionary.tsx       # композиция: дерево-таблица статей (общая ДДС/БДР)
    article-form.tsx             # client-форма статьи на useActionState (общая ДДС/БДР)

lib/
  domain/
    reference/
      articles.ts   +  articles.test.ts
      bank-account.ts + bank-account.test.ts

prisma/
  schema.prisma                  # + миграция reference-dictionaries
  seed.ts                        # + тестовые данные справочников

tests/
  e2e/
    reference.spec.ts
```

Соответствие правилам структуры (CLAUDE.md):

- `app/<раздел>/` — каждый справочник самостоятелен; форма, специфичная разделу
  (счета), лежит рядом с маршрутом, как `transaction-form.tsx`.
- `components/` (вне `ui/`) — только переиспользуемое двумя маршрутами (дерево
  статей и форма статьи). Форма статьи — чистая композиция: `kind`, server
  action и список групп-родителей приходят пропсами от страницы.
- `lib/domain/reference/` — чистая логика с unit-тестами рядом; без React и
  Prisma.
- `components/ui/` руками не трогаем — добавляем shadcn-примитивы через
  `npx shadcn add select checkbox textarea badge`.

## 4. Модель данных (Prisma)

Статьи ДДС и БДР — одна модель, различаются `kind`. Тип «поступление/выбытие»
(ДДС) и «доход/расход» (БДР) семантически — один и тот же приток/отток денег,
поэтому один enum `ArticleFlow`, по-разному подписанный в UI.

```prisma
enum ArticleKind { CASHFLOW  PNL }
enum ArticleFlow { INFLOW  OUTFLOW }

model Article {
  id          String       @id @default(cuid())
  kind        ArticleKind
  name        String
  code        String?
  flow        ArticleFlow?          // тип; у групп/папок пусто
  isGroup     Boolean      @default(false)
  description String?
  parentId    String?
  parent      Article?     @relation("ArticleTree", fields: [parentId], references: [id])
  children    Article[]    @relation("ArticleTree")
  isActive    Boolean      @default(true)   // архив = false
  createdAt   DateTime     @default(now()) @db.Timestamptz(3)

  @@index([kind, parentId])
  @@map("articles")
}

model BankAccount {
  id            String   @id @default(cuid())
  name          String
  accountNumber String
  bankName      String
  bankBic       String
  currency      String   @default("RUB")
  organization  String
  isActive      Boolean  @default(true)
  createdAt     DateTime @default(now()) @db.Timestamptz(3)

  @@map("bank_accounts")
}
```

Миграция: `npx prisma migrate dev --name reference-dictionaries`.
Недеструктивная — только новые таблицы и enum'ы, существующие данные не
затрагиваются.

Замечание на будущее (синк с 1С): при появлении интеграции добавим nullable
`externalUid String? @unique` для инкрементального upsert по UID из 1С. Сейчас
не вводим (YAGNI), но мягкое удаление (`isActive`) уже совместимо с моделью
«помеченных на удаление» в 1С.

## 5. Доменная логика (`lib/domain/reference/`, чистая, unit-тесты рядом)

### `articles.ts`

- `buildArticleTree(items)` — из плоского списка строит дерево (узлы с
  `children` и вычисленной `depth` для отступов в таблице).
- Сортировка узлов: по `code` (натурально), затем по `name` (локаль `ru`).
- Валидация (используется в server actions):
  - `name` непустой;
  - родитель (если задан) существует, того же `kind` и является группой
    (`isGroup=true`);
  - защита от циклов: нельзя назначить родителем себя или своего потомка;
  - конечная статья (`isGroup=false`) требует `flow`; у группы `flow` пустой.

### `bank-account.ts`

- Валидаторы: БИК — ровно 9 цифр; номер счёта — ровно 20 цифр; непустые
  `name`, `bankName`, `organization`; `currency` — непустой код (по умолчанию
  `RUB`).

Все функции — чистые, детерминированные, покрыты тестами (TDD).

## 6. Мутации (server actions)

Паттерн из `app/transactions/actions.ts`: сигнатура
`(prevState: FormState, formData: FormData) => Promise<FormState>`; ожидаемые
ошибки валидации возвращаются как `{ error }` и показываются формой через
`useActionState`; неожиданные сбои ловит `app/error.tsx`; после успеха —
`revalidatePath`.

Действия по каждому справочнику:

- **Статьи** (`cashflow-items/actions.ts`, `pnl-items/actions.ts`): `kind`
  фиксирован на уровне модуля. `create`, `update`, `archive`/`unarchive`.
  Перед записью — валидация из `lib/domain/reference/articles.ts` (для проверки
  родителя/циклов action подгружает нужные записи того же `kind`).
- **Банковские счета** (`bank-accounts/actions.ts`): `create`, `update`,
  `archive`/`unarchive`; валидация из `lib/domain/reference/bank-account.ts`.

## 7. UI (только shadcn в рамках темы)

- **Список статей** — server component: читает `Article` по `kind` из Prisma
  (`lib/db.ts`), передаёт в `<ArticleDictionary>`. Дерево выводится плоской
  таблицей с отступом по `depth`. Переключатель «показать архивные».
  Тип и признак группы — бейджами (`badge`).
- **Форма статьи** (`article-form.tsx`, client, `useActionState`): наименование,
  код, тип (`select`; для ДДС подписи «поступление/выбытие», для БДР
  «доход/расход»), «это группа» (`checkbox`; при включении тип скрывается),
  родитель (`select` среди групп того же `kind`), описание (`textarea`).
- **Список счетов** — server component + таблица: название, номер, банк (БИК),
  валюта, организация, статус (активен/архив).
- **Форма счёта** (`bank-account-form.tsx`, client, `useActionState`):
  название, номер счёта, банк, БИК, валюта, организация-владелец.
- **Витрина `app/reference/page.tsx`** — три карточки-ссылки (shadcn `card`),
  как на главной. На главной (`app/page.tsx`) добавляем ссылку «Справочники».
- Иконки — только `lucide-react`. Цвета/отступы — токены темы, без хардкода.

## 8. Источник данных и запрос к 1С (отложено)

### Текущее состояние доступа

Проверено на 2026-07-16: подключения к 1С в finflow нет.

- В `.env` только `DATABASE_URL` (локальный dev-Postgres);
  `DATABASE_URL_PROD_RO` не выдан.
- Нет `lib/integrations/`, `lib/sync/`, нет драйвера `mssql`.
- Согласованная в спеке «Заявки на оплату» (§4) схема чтения через
  **DWH/DEEONE (SQL Server, SSH host02 → ai01)** — план, не реализована.
  1С REST API (`http://192.168.79.250:4480/api/1crm/...`) — только на запись.

Поэтому в этой итерации справочники ведутся в finflow на seed-данных, а синк —
будущая фича.

### Черновик запроса программистам 1С (для будущей интеграции)

> **Зачем.** В finflow добавлен раздел «Справочники»: Статьи ДДС, Статьи БДР,
> Банковские счета. Хотим показывать их из 1С как источника истины (только
> чтение), синхронизацией — по образцу модуля «Заявки на оплату».
>
> **Просим:**
> 1. Подтвердить, реплицируются ли эти справочники в DWH DEEONE. Если да — дать
>    имена таблиц/представлений и доступ на чтение существующим каналом
>    (SSH host02 → ai01). Если нет — добавить их в витрину или открыть GET-
>    эндпоинты в REST API 1С.
> 2. Поля по каждому справочнику:
>    - Статьи ДДС («Статьи движения денежных средств»): `UID (GUID)`, `Код`,
>      `Наименование`, `Родитель (UID)`, `ЭтоГруппа/IsFolder`,
>      `ВидДвижения/Тип`, `ПометкаУдаления`, `Описание`.
>    - Статьи БДР (справочник статей бюджета доходов/расходов — уточнить точное
>      имя объекта): те же поля, тип = доход/расход.
>    - Банковские счета: `UID`, `Наименование`, `НомерСчета`, `Банк.Наименование`,
>      `Банк.БИК`, `Валюта`, `Организация-владелец`, `ПометкаУдаления`.
> 3. Протокол доступа (таблицы/вьюхи DWH, REST GET или read-only реплика).
>    Критично: только чтение, стабильные `UID`, отдача помеченных на удаление.
> 4. Ориентир по частоте изменений справочников (для расписания синка).
>
> Контакт по деталям со стороны finflow: Минас (разработчик).

Отправка запроса — вне рамок этой фичи; передаётся Минасу/команде 1С вручную.

## 9. Seed (тестовые данные)

`prisma/seed.ts` наполняет справочники демо-деревом и счетами (данные создаются
идемпотентно; e2e от seed не зависят — см. §10). Ориентировочный состав:

- **Статьи ДДС:** группа «Операционная деятельность» → «Поступления от
  покупателей» (поступление), «Оплата поставщикам» (выбытие); группа
  «Финансовая деятельность» → «Кредиты и займы» (поступление).
- **Статьи БДР:** группа «Доходы» → «Выручка» (доход); группа «Расходы» →
  «Зарплата» (расход), «Аренда» (расход).
- **Банковские счета:** 2–3 счёта в RUB по разным организациям и банкам
  (например, Сбербанк, Т-Банк) с валидными по формату БИК/номером.

## 10. Тестирование (гибрид по слоям)

- **Unit (Vitest, TDD)** — вся логика `lib/domain/reference/`: построение
  дерева и глубины, сортировка, циклы, «родитель — только группа того же
  kind», «лист требует тип»; валидаторы БИК (9 цифр) и счёта (20 цифр).
- **E2e-смоук (Playwright, `tests/e2e/reference.spec.ts`)** — главный сценарий:
  открыть «Справочники» → в ДДС создать группу и внутри неё статью, увидеть их
  деревом → создать банковский счёт → увидеть в списке. Данные создаются внутри
  теста, без зависимости от seed.
- Unit-тесты на React-компоненты/страницы не пишем (по правилам проекта — их
  роль выполняет e2e).
