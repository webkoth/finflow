# Спека: авторизация и роли в приложении

Дата: 2026-07-15
Статус: дизайн утверждён (брейншторминг с Минасом), готово к плану реализации.
Источники: спека заявок `2026-07-14-payment-requests-design.md` (§3 — ролевая
модель), спека окружения `2026-07-13-team-environment-design.md` (§14 —
«фича на потом»), рабочая авторизация старого приложения (репо `fin`:
Flask, cookie `fin_session`, роли, хеши паролей).

## 1. Контекст и проблема

Оба контура finflow закрыты nginx basic auth с общим логином/паролем: все
пользователи неразличимы, права не разграничены, `author` в комментариях
бухгалтера — свободная строка. Модули заявок и светофора уже содержат
действия с разной ценой ошибки (согласование платежей, настройки порогов) —
им нужны настоящие пользователи и роли.

## 2. Границы

### Входит

1. Логин/пароль с cookie-сессиями (без внешних провайдеров).
2. Три роли и enforcement прав в UI и server actions.
3. Управление пользователями (страница собственника) и смена своего пароля.
4. Привязка авторства: `ExecutionComment.authorId`; требование к
   `PaymentOrderDispatch.confirmedBy` (план 05).
5. Bootstrap первого собственника, демо-пользователи в seed, адаптация e2e.

### Не входит (осознанно)

- OAuth/SSO, e-mail-восстановление пароля (почтовой инфраструктуры нет —
  пароль сбрасывает собственник).
- Полный аудит действий («кто согласовал заявку», журнал входов) — кандидат
  в бэклог.
- Гибкая матрица прав роль×модуль — YAGNI, роли фиксированные.
- 2FA.

## 3. Ключевые решения

| Тема | Решение |
|---|---|
| Механизм | Собственная cookie-сессионная авторизация, **ноль новых npm-зависимостей**: `node:crypto` scrypt + сессии в PostgreSQL |
| Роли | Три: `owner` (собственник/руководитель), `accountant` (бухгалтер), `viewer` (читатель) — как в спеке заявок §3 |
| nginx basic auth | Убирается на **обоих** контурах после релиза фичи (решение Минаса; риск открытого dev-контура озвучен и принят) |
| Пользователи | Заводит собственник на `/settings/users` с временным паролем; каждый может сменить свой пароль |
| Первый owner | Ленивый bootstrap: пустая таблица + пара из `APP_BOOTSTRAP_LOGIN/PASSWORD` на `/login` → создаётся owner |
| Пароли | scrypt (соль 16 байт), `timingSafeEqual`, минимум 8 символов |
| Сессии | Токен 32 байта; в БД — SHA-256-хеш токена; cookie httpOnly/`sameSite=lax`/`secure` (prod); 30 дней со скользящим продлением |
| Авторство | FK + строковый снапшот имени (история не ломается при переименовании/удалении) |

## 4. Модель данных (Prisma)

- **`User`**: `login @unique` (нижний регистр), `name`, `role`
  (enum `owner | accountant | viewer`), `passwordHash` (`salt:hash`, hex),
  `isActive Boolean @default(true)`, `createdAt`.
- **`Session`**: `tokenHash @unique` (SHA-256 от токена — дамп БД не даёт
  живых сессий), `userId` (FK, onDelete: Cascade), `expiresAt`, `createdAt`.
- **`ExecutionComment`** (расширение): `authorId String?` (FK на `User`,
  nullable — старые записи остаются со строкой `author`); `author` теперь —
  снапшот имени на момент записи, форма его не спрашивает.

## 5. Механика

- **Хеширование** (`lib/auth/passwords.ts`): scrypt из `node:crypto`
  (N=16384, r=8, p=1), соль 16 байт `randomBytes`, сравнение
  `timingSafeEqual`. Unit-тесты рядом.
- **Логин** (server action): нормализация логина → проверка пары →
  `Session` (токен `randomBytes(32)`, в БД — его SHA-256) → cookie
  `finflow_session`. При любой ошибке — одинаковое «Неверный логин или
  пароль». Попутно удаляются истёкшие сессии (`deleteMany` по `expiresAt`).
- **Bootstrap**: если пользователей нет вообще и пара совпадает
  с `APP_BOOTSTRAP_LOGIN`/`APP_BOOTSTRAP_PASSWORD` — создать owner и войти.
- **Текущий пользователь**: `getCurrentUser()` (React `cache()` — один
  запрос на рендер): cookie → сессия join user → `null`, если истекла или
  `isActive = false` (деактивация действует мгновенно). Скользящее
  продление: если сессии больше суток — `expiresAt` сдвигается на 30 дней.
- **`middleware.ts`**: только быстрый редирект на `/login?callbackUrl=…`
  при отсутствии cookie (Prisma в edge недоступна); настоящая проверка —
  `getCurrentUser()` в страницах и `requireRole()` в server actions.
- **Выход**: удаление сессии из БД + очистка cookie.
- **Смена пароля**: удаляет все прочие сессии пользователя.

## 6. Роли и enforcement

Двухслойный enforcement: UI скрывает недоступные элементы; каждый server
action проверяет роль (`requireRole([...])` → `{ error: "Недостаточно
прав" }`). Страницы `owner`-only отдают 404 остальным.

| Действие | owner | accountant | viewer |
|---|---|---|---|
| Просмотр всех страниц и отчётов | ✓ | ✓ | ✓ |
| Согласовать / отклонить (карточка и массово) | ✓ | — | — |
| Комментарии бухгалтера | ✓ | ✓ | — |
| Статьи ДДС, отправка платёжек (план 05) | ✓ | ✓ | — |
| Настройки светофора | ✓ | — | — |
| Пользователи (`/settings/users`) | ✓ | — | — |
| Кнопка «Обновить» (ручной синк) | ✓ | ✓ | ✓ |

Матрица — чистая функция `lib/domain/permissions.ts`
(`can(role, action): boolean`), unit-тесты на каждую клетку; UI и actions
пользуются только ею. `POST /api/jobs/sync` остаётся на cron-секрете
(механизм для планировщика, не для людей).

## 7. Экраны

1. **`/login`** — логин + пароль, ошибка одним сообщением, редирект на
   `callbackUrl` после входа.
2. **Шапка приложения** (root layout, на всех страницах): имя пользователя,
   бейдж роли, «Выйти».
3. **`/settings/users`** (owner): таблица (логин, имя, роль, статус,
   создан); создать пользователя (временный пароль вводит owner и передаёт
   лично), сменить роль, сбросить пароль, деактивировать/активировать.
   Защита от самострела: нельзя деактивировать себя; нельзя понизить или
   деактивировать последнего активного owner.
4. **`/settings/password`** — старый пароль + новый дважды (минимум 8
   символов).

## 8. Авторство

- Форма комментария бухгалтера перестаёт спрашивать «Автор»: пишутся
  `authorId` из сессии + `author` (снапшот имени). Старые записи
  (`authorId = null`) отображаются как раньше.
- **Требование к плану 05**: `PaymentOrderDispatch` подтверждения писать
  той же схемой — `confirmedById` (FK) + снапшот имени.
- Право комментирования: owner и accountant (см. §6).

## 9. Инфраструктура и предпосылки

- **env**: `APP_BOOTSTRAP_LOGIN`, `APP_BOOTSTRAP_PASSWORD`
  (+ `APP_BOOTSTRAP_NAME` опционально) — только для первого входа;
  в `.env.example` с комментарием.
- **Seed (dev)**: три демо-пользователя — `e2e-owner`, `e2e-accountant`,
  `e2e-viewer` с известными паролями (для e2e и песочницы).
- **Снятие basic auth** (вне кода, делает разработчик): после релиза фичи
  в prod убрать basic auth из nginx-конфигураций обоих контуров.
  До этого — временный двойной вход. `/status` (HTTP-проверка) начнёт
  получать 200/302 вместо 401 — поведение команды не ломается.
- Спека окружения §4/§14 после релиза считается изменённой в этой части
  (basic auth → внутренняя авторизация).

## 10. Тестирование

- **Unit**: `lib/auth/passwords.ts` — roundtrip хеш/проверка, неверный
  пароль → false, уникальная соль (два хеша одного пароля различны),
  формат `salt:hash`; `lib/domain/permissions.ts` — вся матрица §6.
- **E2e**: хелпер `loginAs(page, role)`; сценарии — вход/выход; неверный
  пароль; редирект неавторизованного на `/login` и обратно; viewer не
  видит кнопок согласования, а прямой вызов action получает «Недостаточно
  прав»; accountant не открывает `/settings/users` (404); owner создаёт
  пользователя → тот входит; смена пароля (старая сессия второго браузера
  умирает); деактивация отрубает вход. Все существующие e2e дополняются
  логином.

## 11. Пополнение бэклога (по итогам этого дизайна)

- Аудит действий: кто/когда согласовал, отклонил, изменил настройки;
  журнал входов.
- Привязка согласований к пользователю (`approvedBy` на заявке) — вместе
  с аудитом.
