# Спека: отчёт «БДР Маркетплейсы»

Дата: 2026-07-15
Статус: дизайн утверждён (брейншторминг с Минасом), готово к плану реализации.
Источники: `fin/finflow/03-bdr-marketplaces.md` (статьи и формулы, сверки),
`fin/finflow/04-cogs-and-delivery.md` (себестоимость, доставка),
`fin/finflow/05-lower-blocks.md` (нижние блоки), рабочая реализация
`fin/finflow-app/bdr.py` + `BdrMarketplaces.jsx`, БД-кэши
`bdr_wb_db_cache.json` / `bdr_ozon_db_cache.json`.

Общая инфраструктура (scope `reports` синка, механика Excel-загрузок,
право `manage_report_settings`, зависимость `exceljs`) — введена спекой
«ИУ WB план-факт» (`2026-07-15-iu-wb-report-design.md`) и переиспользуется.

## 1. Контекст и проблема

Недельный P&L по маркетплейсам: 6 кабинетов (WB: ИП Бобровская, ТОРИ,
Русбубон; OZON: LARETTO, ШАПКИ, LRTT), статьи от выручки до валовой прибыли
+ общефирменные нижние блоки (opex, прибыль, распределение, баланс).
В старом приложении верх уже мигрирован на БД (сходится с ориентиром до
рубля), нижние блоки — из ориентира/ФИН.МОД. Переносим на архитектуру
finflow с сохранением выверенной логики.

## 2. Границы

### Входит

1. **Верх (кабинетная таблица)**: статьи WB и OZON из БД по неделям,
   cogs по прайс-листу 1С, фильтры каналы × кабинеты, KPI, график,
   режимы «Сравнение недель» / «Все недели».
2. **Ориентир-fallback**: загрузка ориентира (xlsx) в UI; недели без
   БД-данных берутся из ориентира с пометкой источника.
3. **Нижние блоки**: входы (opex, балансы, прочие) из загруженного
   ориентира (значения ячеек уже вычислены Excel), формулы (опер./чистая
   прибыль, распределение, коэффициенты) пересчитываются в коде от живой
   валовой из БД.
4. Настройки: загрузка ориентира, исключённые бренды-комиссионеры.

### Не входит (осознанно)

- **Активация нижних входов через API 1С** (opex, балансы, доставка
  начислением) — методы у Кати есть, но в систему не заведены
  (предпосылка не закрыта); войдут отдельной итерацией. Строка «Доставка
  до складов» в v1 показывает значение ориентира (сейчас пусто) с пометкой.
- **Расщепление OZON «выручка/баллы»** — в БД-транзакциях его нет
  (`accruals_for_sale` = сумма); `revenue` включает баллы, строки «Баллы»
  нет (ограничение зафиксировано, ждёт данных).
- Факт. себестоимость партий — принят прайс-лист ±1–4% (решение
  зафиксировано ранее).

## 3. Ключевые решения

| Тема | Решение |
|---|---|
| Верх таблицы | Из БД (порт выверенных запросов `bdr_wb_lines_from_db` / `bdr_ozon_lines_from_db`); ориентир — fallback для недель без БД-данных (лаг OZON ~10 дней), с пометкой источника в UI |
| Нижние блоки | Входы из ориентира (вычисленные значения ячеек), формулы — в коде от живой валовой |
| Ориентир | Загрузка xlsx в UI (механика загрузок ИУ), история; парсер читает лист «БДР» по координатам из доки |
| Cogs | Нетто × прайс-лист 1С («с доп.расходами» / закупочная) по коду 1С из `sa_name`, через `OPENQUERY(BI.[1cv])` |
| Бренды-комиссионеры | Vontel и НШФ исключаются из WB-статей; список — настройка в БД |
| Хранение | `BdrWeekLine` (БД-статьи) — порт json-кэшей; `BdrOrientirLine` — распарсенный ориентир |
| Синк | Тот же scope `reports` (ночной cron + кнопка), шаги WB / OZON / cogs независимы |
| Недели | Коды ГГНН, ISO Пн–Вс; общий доменный хелпер с отчётом ИУ |

## 4. Архитектура

Переиспользование: mssql-пул и туннель плана 8; `POST /api/jobs/sync?scope=reports`
и cron из спеки ИУ; права из плана 7 (+`manage_report_settings`); `exceljs`.

Поток: DWH (`WBALL.wb_report_detail_by_period`, `OZONALL.ozon_finance_transactions`
(+`_services`), `OPENQUERY(BI.[1cv])` для цен) → `BdrWeekLine`; ориентир-xlsx →
`BdrOrientirLine`; экран = домен `lib/domain/bdr.ts` от (БД-строки + ориентир +
настройки).

## 5. Модель данных (Prisma)

- **`BdrWeekLine`** — БД-статьи: `channel` (`wb | ozon`), `blockId`
  (`ip-bobrovskaya | tori-brands | rusbubon | laretto | shapki | lrtt`),
  `weekCode Int` (ГГНН), `line` (`revenue | commission | logistics |
  storage | ads | acceptance | fines | acquiring | other | cogs`),
  `value Decimal(18,2)`, `syncedAt`.
  `@@unique([channel, blockId, weekCode, line])`.
- **`BdrOrientirLine`** — распарсенный ориентир: `weekCode`, `rowKey`
  (статьи кабинетов `wb:ip-bobrovskaya:revenue`… и строки нижних секций
  `sections:opex_prod`, `sections:balance_inventory`…), `value Decimal(18,2)`,
  `uploadId` (FK).
- **`BdrOrientirUpload`** — `fileName`, `rowsParsed`, `uploadedById` +
  снапшот имени, `createdAt`. Активна последняя загрузка.
- **`BdrSetting`** — `key @unique`, `value String`. Сид:
  `excludedBrands = "Vontel,НШФ"` (применяется в WB-запросах синка).

Маппинг аккаунтов на блоки — константы домена: WB `1→tori-brands,
2→ip-bobrovskaya, 3→rusbubon`; OZON `1→shapki, 2→laretto, 3→lrtt`.

## 6. Доменная логика (`lib/domain/bdr.ts`)

Чистые функции; unit-тесты на контрольных суммах W24 из доки (сверено
с ориентиром до рубля: Бобровская выручка 51 562 771, комиссия −7 638 211,
грязная 34 170 285, cogs −17 578 999, валовая 16 591 287; ТОРИ и Русбубон
аналогично).

- **Формулы статей WB** (считаются в синке из строк реализации): выручка =
  Σ retail_amount(Продажа) − Σ(Возврат); комиссия = pay_net − выручка;
  логистика/хранение/реклама/приёмка/штрафы = −Σ соответствующих колонок.
- **OZON residual-подход**: revenue = Σ accruals_for_sale; маппинг услуг
  и операций на logistics/acquiring/ads/storage; `other` = Σ amount −
  (всё замапленное) — замыкает грязную; `gross_rev` = Σ amount.
- **Сборка датасета** `line × channel × blockId × week`: БД переопределяет
  ориентир на неделях, где есть `BdrWeekLine`; у каждой недели признак
  источника (`db | orientir`).
- `gross_rev` = сумма статей; `gross_profit` = gross_rev + cogs + delivery
  (расходы отрицательные).
- **Нижние блоки**: opex/балансы/прочие — из ориентира; операционная
  прибыль = валовая (по всем кабинетам, из живого датасета) + opex; чистая
  прибыль = опер. + проч. доходы + проч. расходы; распределение ЧП
  (безопасность 5%, реинвест 60%, страховка 6%, сверхдоход >6% рент.,
  фонд СУ, дивиденды); коэффициенты (ROE нед./год, наценка, доля комиссий,
  маржа, рентабельность, оборачиваемость, леверидж).
- **Недели ГГНН** — общий хелпер с ИУ (`weekCode(date)`, `weekRange`,
  сравнительные недели: прошлая / месяц назад / год назад).

## 7. Экраны

### 7.1 `app/reports/bdr` — отчёт

- Фильтры-чипы: каналы (WB/Озон) × кабинеты; таблица = сумма выбранных;
  нижние секции от фильтров не зависят.
- KPI-карточки и линейный график (выручка / грязная / валовая) — shadcn
  charts, фильтр периода графика.
- Режимы: «Сравнение недель» (отчётная / прошлая / месяц назад / год назад)
  и «Все недели».
- Таблица статей с раскрываемыми подстатьями; недели помечены источником
  (БД / ориентир); строка свежести (scope reports) + «Обновить».
- Нижние секции: опер. расходы и прибыль, распределение ЧП, баланс
  и эффективность; строка «Доставка до складов» с пометкой «начисление
  появится с API 1С».

### 7.2 `app/reports/bdr/settings` — настройки (owner)

- Загрузка ориентира (история загрузок, «распарсено N строк»).
- Исключённые бренды (строка-настройка).
- Право `manage_report_settings`; 404 остальным.

## 8. Синк, ошибки, деградация

- Тот же `scope=reports`: шаги `bdr_wb` (статьи из реализации), `bdr_ozon`
  (транзакции + услуги), `bdr_cogs` (кол-во × цены из 1С) — независимы,
  результат по шагам в `SyncRun`.
- Окно: последние 10 ISO-недель перезаписываются каждым запуском
  (корректировки и лаг OZON), `full=1` — вся история.
- Лаг OZON (~10 дней): свежие недели без БД-данных показываются из
  ориентира с пометкой — сообщение BD-команде о лаге остаётся в силе.
- `OPENQUERY` недоступен (нет прав) → шаг cogs падает с понятной ошибкой,
  статьи WB/OZON живут; валовая помечается «без cogs».
- Ориентир не загружен → отчёт работает только на БД-неделях; нижние
  секции — «нет данных ориентира».

## 9. Тестирование

- **Unit (`lib/domain/bdr.ts`)**: формулы WB-статей и OZON-residual на
  синтетических строках; merge БД/ориентир (переопределение, пометка
  источника); gross_rev/gross_profit; нижние формулы (опер./чистая,
  распределение, коэффициенты) на контрольных цифрах W24; хелпер ГГНН
  (2624 = 2026-W24, границы года).
- **Unit (парсер ориентира)**: эталонный xlsx-фрагмент из fixtures
  (коды недель в строке 3, блоки кабинетов, нижние секции, пропуск
  блока «долей»).
- **Интеграционные**: синк на замоканных ответах DWH (upsert, окно 10
  недель, независимость шагов, применение excludedBrands).
- **E2e**: отчёт на fixtures (снапшоты реальных кэшей) — таблица
  с кабинетами, фильтр кабинета меняет суммы, переключение режимов,
  подстатьи раскрываются, нижние секции видны; настройки — загрузка
  ориентира; 404 для не-owner.

## 10. Этапность реализации (детали — в плане)

1. Prisma + домен (формулы, merge, ГГНН) + fixture-данные + экран верха
   (таблица, фильтры, KPI, график, режимы).
2. Ориентир: парсер + загрузка + fallback-merge + нижние блоки с формулами.
3. Боевой синк: WB + OZON статьи через пул плана 8, сверка с контрольными
   суммами W24.
4. Cogs через `OPENQUERY(BI.[1cv])` (после подтверждения прав) + валовая.

## 11. Предпосылки и открытые вопросы

1. **SQL-логин** с правами на `marketplaces` — общая предпосылка с планом 8.
2. **Права на `OPENQUERY`** (linked servers `192.168.79.250`/BI, `OLAP`)
   у нового SQL-логина — добавить в запрос BD-команде. *(Блокирует только
   cogs; статьи работают без него.)*
3. **Лаг загрузки OZON-финансов** (~10 дней) — вопрос BD-команде открыт;
   до решения — ориентир-fallback.
4. **API 1С** (opex, балансы, доставка начислением) — методы есть у Кати,
   не заведены; активация нижних входов — следующая итерация.
5. **Сверка после порта**: недельные суммы против действующего отчёта
   (W24-эталон в доке, допуск 0 для статей, ±1–4% для cogs).

## 12. Пополнение бэклога

- Активация нижних входов через API 1С (opex, балансы, прочие доходы/расходы).
- Доставка до складов начислением (API 1С) — отдельная строка нижних блоков.
- Расщепление OZON «выручка/баллы» при появлении данных в БД.
- Перевод cogs на факт. себестоимость партий, если ±1–4% перестанет
  устраивать (нужна карта метаданных 1С или API 1С).
