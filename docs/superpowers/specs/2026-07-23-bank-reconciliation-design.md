# Сверка расчётных счётов: выписка ↔ 1С ↔ заявки

Дата: 2026-07-23
Статус: дизайн утверждён, готов к плану реализации

## Задача

Ежедневный контур контроля расчётных счётов. Независимая банковская выписка
(эталон) сверяется с тем, что казначей провёл в 1С, и с заявками на оплату.
Цель №1 — **исключить ручные правки казначея**: изменённые суммы, обороты,
остатки и подменённых получателей. Эталон должен быть независим от казначея.

Процесс запускается автоматически каждый рабочий день в фиксированное время;
результат накапливается как история (не разовый аудит) и виден собственнику
в приложении finflow.

## Что проверяем

Три плоскости контроля вокруг одного дня по каждому счёту:

**Плоскость 1 — счёт (остатки и обороты):**
выписка ↔ остаток из 1С (`AccountBalance`) + обороты по счёту из 1С.
Сверяем 4 числа: остаток на начало, остаток на конец, оборот-дебет,
оборот-кредит; плюс тождество `начало + кредит − дебет = конец`.

**Плоскость 2 — заявка → списание → выписка** (по каждому списанию за день):
1. **Заявка исполнена** — у одобренной заявки (`PaymentRequest`) есть фактическое
   списание с `ДокументОснование` = эта заявка, на её сумму.
2. **Оплатили кому хотели** — получатель и реквизиты совпадают в трёх точках:
   заявка (`partnerInn`/`partnerName`) = списание в 1С (контрагент + счёт) =
   выписка (получатель + ИНН + счёт).

## Границы

**Входит:** доменное ядро сверки, история в БД (append-only), UI (бейджи на
остатках + экран истории + метка на заявках), ежедневный запуск, парсинг
выписки, слой источника выписки (авто из банка + ручной файл).

**Не входит сейчас:**
- Автоматический вход в банк-клиенты и обход СМС/2FA — **принципиально не
  делаем**. Независимый доступ обеспечивается отдельным банковским
  пользователем «только просмотр» (организационный шаг собственника), а не
  обходом аутентификации.
- Построчное хранение каждого платежа. Глубина хранения — **итоги + расхождения**
  (сверка построчная в момент прогона, в БД оседают только сводки и найденные
  расхождения).

## Архитектура

Три части по слоям finflow:

1. **Доменное ядро** — `lib/domain/reconciliation/`. Чистые функции без сети и
   Prisma, обязательные unit-тесты (Vitest, TDD). Вход: нормализованные выписка,
   движения 1С, заявки. Выход: итоги по счёту + список расхождений.
2. **Хранилище истории** — новые таблицы Postgres (Prisma), append-only.
3. **Интеграции и оркестрация** — `lib/integrations/bank-statement/` (парсинг +
   источник), расширение `lib/integrations/one-c-odata` (движения), оркестрация
   в `lib/sync/run-reconciliation.ts`, джоб `app/api/jobs/reconcile`.
4. **UI** — `app/reconciliation/`, бейджи на дашборде и в справочнике счётов,
   метка на экране заявок.

### Поток данных

```
cron (рабочий день, фикс. время)
  → POST /api/jobs/reconcile (x-sync-secret)
    → lib/sync/run-reconciliation.ts (по каждому счёту):
        StatementSource.getStatement(account, day)   ← авто-банк | файл
        oneC.getAccountMovements(account, day)        ← Document_РасходСоСчета + поступления
        prisma.accountBalance (остаток 1С)
        prisma.paymentRequest (одобренные заявки дня)
      → lib/domain/reconciliation (чистая сверка)
      → prisma: ReconciliationRun + AccountResult + Discrepancy (append-only)
  → бейджи и экран истории читают сохранённое
```

## Модель данных (Prisma, append-only)

Суммы — `BigInt` (обороты легко превышают лимит Int ±21,4 млн ₽), показ через
`formatMoneyBig`, в клиентские компоненты — строкой (правило finflow).

**`ReconciliationRun`** — прогон за день:
- `id`, `runAt @db.Timestamptz(3)`, `periodStart`, `periodEnd`
- `status`: `matched` | `discrepancy` | `no_data`
- `trigger`: `cron` | `manual`
- дочерние: `accountResults`, `discrepancies`

**`ReconciliationAccountResult`** — итог по счёту в прогоне:
- `runId`, `accountUid` (связь с банковским счётом 1С), `bankName`, `accountNumber`,
  `currency`
- эталон: `stmtOpeningMinor`, `stmtClosingMinor`, `stmtDebitMinor`, `stmtCreditMinor`
  (BigInt, nullable если выписки нет)
- 1С: `onecClosingMinor`, `onecDebitMinor`, `onecCreditMinor` (BigInt, nullable)
- `status`: `matched` | `discrepancy` | `no_data` | `source_error`
- источник: `sourceType` (`bank_api` | `manual_file`), `sourceStatus`
  (`ok` | `error`), `sourceError` (String?)
- первоисточник: `statementFileName`, `statementSha256` (для детекции подмены)

**`ReconciliationDiscrepancy`** — по строке на расхождение:
- `runId`, `accountResultId?`, `requestUid?` (ссылка на заявку)
- `type`: `opening_balance` | `closing_balance` | `debit_turnover` |
  `credit_turnover` | `balance_identity` | `recipient_mismatch` |
  `request_not_executed` | `payment_without_request` | `amount_mismatch`
- `expected` (эталон), `actual` (1С), `amountMinor` (BigInt?), `detail` (String —
  где именно разошлось, напр. «заявка↔выписка»)
- разбор (**единственное редактируемое руками**): `resolutionStatus`
  (`new` | `reviewed` | `accepted`), `resolvedById?`, `resolvedAt?`, `note?`

Enum'ы объявляются в схеме рядом с существующими (`ExecutionStatus` и т.п.).

Миграция — `npx prisma migrate dev --name bank-reconciliation`. Не деструктивная
(только новые таблицы), существующие данные не трогает.

## Логика сверки (`lib/domain/reconciliation/`)

Детерминированные правила, никаких «примерно».

**Плоскость 1 — остатки и обороты:**
- **Точное равенство в копейках** (целочисленный `BigInt`, без float-допусков).
  Копейка расхождения = сигнал.
- Сверка в валюте счёта, без пересчёта в рубли (пересчёт — только для показа итога).
- Нет данных 1С по счёту за день → статус `no_data` (не путать «не синкнулось» с
  «подменили»).

**Плоскость 2 — заявка → списание → выписка:**
- **Матч заявка↔списание** по `ДокументОснование` (жёсткая ссылка 1С). Списание
  без основания → `payment_without_request`. Одобренная заявка с плановой датой
  ≤ сегодня без списания → `request_not_executed`.
- **Сумма:** списание = заявка в копейках. Частичная оплата (списание < заявки) —
  **не расхождение**, а пометка «исполнено частично» с остатком. Превышение
  (списали больше заявки) → `amount_mismatch` (красный флаг).
- **Получатель — матч по приоритету:**
  1. **ИНН** — главный ключ (`partnerInn` заявки vs ИНН в выписке vs контрагент 1С).
  2. ИНН пуст/не совпал → сверка по **номеру счёта получателя**.
  3. Нет ни ИНН, ни счёта → сверка по **нормализованному названию** (регистр,
     кавычки, орг-форма «ООО/ИП» отброшены) — **слабое** совпадение, помечается
     «проверить вручную», не «ок».
  - Несовпадение на любом стыке → `recipient_mismatch` с указанием стыка
    (заявка↔1С / 1С↔выписка / заявка↔выписка).

**Статус прогона:** `matched` — все счета сошлись и все списания прошли обе
проверки; `discrepancy` — есть хоть одно расхождение; `no_data` — нечего сверять.
Наличие `source_error` хотя бы по одному счёту не даёт общий `matched`.

## Слой источника выписки

**Абстракция `StatementSource`** — интерфейс «дай выписку по счёту за день»,
за ним провайдеры. Ядро сверки и хранение от источника не зависят.

- **`manual-file`** — загрузка файла на экране + чтение из выделенной папки.
  Делается сразу, работает с первого дня.
- **`bank-api`** — авто-подключение к банк-клиенту под read-only пользователем
  собственника. Подключается по мере готовности доступов, по одному банку.

**Сбои автоподключения исключены явно:**
- Не удалось получить выписку (нет связи, таймаут, ошибка банка, файл не пришёл)
  → статус счёта `source_error` (**не** `matched`, **не** `no_data`).
- Бейдж при `source_error` — **красный «Выписка не получена»**, никогда не зелёный.
  Сбой соединения физически не может выглядеть как «проверено».
- В прогоне сохраняется `sourceType` + `sourceStatus`/`sourceError` — сбои видны
  в истории, не теряются.
- Пока авто по счёту не отдало выписку — можно **дозагрузить файл вручную**, счёт
  досверится. Авто и ручной режим сосуществуют по каждому счёту.

**Парсинг выписки** — `lib/integrations/bank-statement/`, формат
**`1CClientBankExchange`** (kl_to_1c) — стандартный текстовый экспорт всех 5 банков
(Сбер / ВТБ / Альфа / Азия-Инвест / WB). Извлекаем: расчётный счёт,
`ДатаНачала`/`ДатаКонца`, `НачальныйОстаток`/`КонечныйОстаток`, по каждому
документу — плательщик/получатель, ИНН, счёт, сумму. Режимы `fixture`/`file`
через env — без сети в dev/e2e. Другой формат банка — отдельный парсер позже,
ядро сверки от формата не зависит.

**Движения 1С** — расширение `lib/integrations/one-c-odata` методом «движения по
счёту за период» (`Document_РасходСоСчета` + поступления), режимы fixture/http.

### Зависимости и риски источника (важно)

- Единого способа читать выписку из всех 5 банков нет: у каждого свой доступ
  (API / сертификат / протокол), у Азия-Инвеста и WB публичного API может не быть.
  Каждый банк-коннектор — отдельная работа со своей авторизацией.
- **Банковские ключи/сертификаты вводит собственник/пользователь — агент
  банковские креды не вводит** (жёсткое правило безопасности).
- Любая новая библиотека под конкретный банк — **только с одобрения разработчика
  (Минас)** (правило finflow).
- **Независимость эталона ровно настолько, насколько независим источник файла.**
  Пока read-only доступа в банках нет и файл даёт казначей, система ловит
  расхождения выписки с 1С, но не ловит подмену самой выписки. Полная
  независимость включается, когда собственник заведёт «только просмотр» в банках.

## UI

Всё на shadcn в рамках темы, иконки `lucide-react`, стили — токены темы.

**1. Бейдж «Проверено» на остатках:**
- Дашборд (карточка «Остатки по счетам») и справочник `reference/bank-accounts` —
  у каждого счёта бейдж по последнему прогону: зелёный **«Проверено ✓ ДД.ММ»**
  (`matched`), красный **«Расхождения: N»** (`discrepancy`, → в прогон), красный
  **«Выписка не получена»** (`source_error`), серый **«Нет сверки»** (`no_data`).
- Итоговая сумма остатков помечается «проверено», только если сошлись все счета.

**2. Экран истории сверок — `app/reconciliation/`:**
- Список прогонов по дням (дата, статус, счетов сошлось, расхождений), свежий сверху.
- Открытие прогона → по каждому счёту: итоги эталон vs 1С (4 числа рядом,
  расхождения красным) + ссылка на файл-первоисточник.
- Секция расхождений: тип, где разошлось, ожидалось/факт, сумма; для заявок —
  ссылка на заявку. Перевод расхождения `new → reviewed → accepted` с примечанием
  и автором (единственное редактируемое).

**3. Заявки** — на существующем экране заявки метка исполнения дополняется
результатом сверки: «исполнена и сверена ✓» / «получатель не совпал» / «частично».

Мутации (разбор расхождений, ручная загрузка) — через server actions с сигнатурой
`(prevState, formData) => Promise<FormState>`, ошибки валидации возвращаются
(не бросаются), после успеха — `revalidatePath`.

## Запуск

Cron на сервере дёргает `POST /api/jobs/reconcile` с `x-sync-secret` каждый
рабочий день в фиксированное время — по образцу `app/api/jobs/sync-reference`.
Оркестрация в `lib/sync/run-reconciliation.ts`. Ручной прогон (`trigger: manual`)
запускается загрузкой файла на экране сверки.

## Тесты

- **Ядро `lib/domain/reconciliation/`** — обязательные unit-тесты (TDD): равенство
  копеек, тождество остатков, матч получателя по ИНН/счёту/названию, частичная
  оплата, все типы расхождений.
- **Парсер `1CClientBankExchange`** — unit-тесты на фикстурах выписок.
- **e2e-смоук** (Playwright) главного сценария: загрузка выписки → прогон сверки
  → расхождение видно на экране и бейджем на остатках. Данные создаются в тесте,
  не из seed.
- Unit-тесты на React-компоненты запрещены (правило finflow) — их роль у e2e.

## Этапность реализации

1. **Ядро + хранилище + ручной файл + UI** — полностью рабочий контур на
   `manual-file`: парсер, доменная сверка, таблицы, экран истории, бейджи, джоб,
   заявочная плоскость. Ценность с первого дня.
2. **`bank-api` коннекторы** — по одному банку, по мере готовности read-only
   доступов и одобрения библиотек Минасом. Абстракция `StatementSource` заложена
   на этапе 1, коннекторы вставляются без переделки.

## Открытые вопросы к реализации

- Точное имя набора поступлений на счёт в OData (аналог `Document_РасходСоСчета`
  для прихода) — проверить запросом при реализации движений.
- Формат выписки по каждому банку: подтвердить, что все 5 отдают
  `1CClientBankExchange`; при отклонениях — отдельный парсер.
- Фиксированное время ежедневного запуска — согласовать с собственником.
