# Спека: синхронизация справочников из 1С (OData)

Дата: 2026-07-21
Статус: дизайн (брейншторминг с Антониной), готов к плану реализации.
Источник требований: диалог со специалистом.
Предшественник: `2026-07-16-reference-dictionaries-design.md` (§8, §11 — синк
был осознанно отложен до появления доступа к 1С).

## 1. Контекст и проблема

Раздел «Справочники» (статьи ДДС, статьи БДР, банковские счета) существует в
finflow с 16 июля, но данные в нём ведутся вручную и наполнены seed'ом.
Целевым источником истины всегда предполагалась 1С — предыдущая спека прямо
зафиксировала это и заранее подготовила модель (мягкое удаление, заметка про
будущий `externalUid`).

21 июля выяснилось, что база 1С `rbb_cut` доступна по сети с машины
разработки: `http://192.168.79.250:1281/rbb_cut` отвечает веб-клиентом
1С:Предприятие 8.5.1.1150 (Apache 2.2.23). Это открывает канал чтения, которого
не было в июле.

Задача: **на постоянной основе подтягивать изменения справочников из 1С в
finflow**.

### Состояние доступа на момент написания

Проверено 2026-07-21:

- OData-адрес `…/rbb_cut/odata/standard.odata/` существует и отвечает
  `401 Unauthorized` с заголовком `WWW-Authenticate: Basic realm="1C:Enterprise 8.5"`.
- Заведён пользователь 1С **«Клод только чтение»**; вход этой учёткой через
  веб-клиент в браузере проходит успешно.
- Тот же логин/пароль на OData даёт 401 в трёх кодировках basic-auth
  (UTF-8, CP-1251, latin1) — то есть дело не в передаче учётных данных.

Вывод: учётные данные верны, не хватает **прав на OData**. Требуются два
действия администратора в Конфигураторе (из пользовательского интерфейса 1С
недоступны):

1. роль с правом «Использование стандартного интерфейса OData» для пользователя;
2. *Администрирование → Состав стандартного интерфейса OData* — включить три
   нужных справочника (по умолчанию состав пуст, OData не отдаёт ничего).

Запрос администратору передан через специалиста. **Реализация от этого не
заблокирована**: разработка и приёмка идут на фикстурах, реальное подключение
проверяется отдельным шагом (§9).

## 2. Принятые решения

Зафиксировано в диалоге со специалистом:

1. **1С — единственный источник истины.** В finflow справочники становятся
   только для чтения; CRUD удаляется.
2. **Объём — три существующих справочника**: статьи ДДС, статьи БДР,
   банковские счета. Контрагенты, организации, валюты, договоры — не входят.
3. **Расписание — раз в сутки ночью**, плюс кнопка «Обновить из 1С» вручную.
4. **Удаление в 1С → архив в finflow**: запись скрывается из основного списка,
   но остаётся в базе и видна переключателем «показать архивные».
5. **Канал — OData напрямую** (вариант A, см. §3).

## 3. Выбор канала доступа

Рассмотрены три варианта.

| Вариант | Суть | Почему не выбран |
| --- | --- | --- |
| **A. OData напрямую** | finflow сам читает `standard.odata` под своей учёткой | **выбран** |
| B. HTTP-сервис в 1С | программисты 1С пишут эндпоинт под нас | зависимость от чужой команды и их сроков; OData даёт то же самое без ожидания |
| C. Через DWH DEEONE | как модуль заявок на оплату | не подтверждено, что справочники вообще реплицируются в DWH (открытый вопрос ещё с июльской спеки); канал SSH не построен |

**A** выбран как единственный, не упирающийся в ожидание других команд:
интерфейс уже отвечает, нужны только права. B остаётся запасным, если
публикация OData окажется невозможной. Смена канала затрагивает один модуль —
клиент (§4), остальная архитектура от неё не зависит.

### Механика «выгружать изменения»

1С по OData не инициирует передачу — она только отвечает на запросы. Кроме
того, справочники 1С, в отличие от документов, как правило не имеют реквизита
«дата изменения», поэтому запрос «что поменялось со вчера» скорее всего
невозможен (проверяется при первом успешном подключении).

Поэтому: finflow забирает **полный снимок** справочника (сотни записей —
дёшево), сравнивает со своим состоянием и записывает **только отличия**.
Наблюдаемый результат тот, что требовался: в finflow приезжают изменения из 1С.
Отличается только способ их вычисления — не 1С сообщает, а finflow сравнивает.

Следствие: задержка появления правки в finflow — до суток (или до нажатия
кнопки). Специалистом принято.

## 4. Архитектура

Ведущий принцип: **получение данных, их сравнение и запись разделены**. Вся
сложность — в сравнении, оно оформлено чистой функцией и тестируется без сети
и без базы.

```
lib/integrations/
  one-c-odata.ts            # клиент OData: basic auth, только GET, постранично
  one-c-odata-fixture.ts    # те же данные из файла — dev и e2e, без сети
lib/domain/reference/
  sync-diff.ts              # ЧИСТАЯ логика сравнения
  sync-diff.test.ts         # unit-тесты (TDD)
lib/sync/
  run-reference-sync.ts     # оркестрация: получить → сравнить → записать → журнал
app/api/jobs/sync-reference/
  route.ts                  # точка входа ночного расписания
app/reference/
  actions.ts                # server action кнопки «Обновить из 1С»
```

Соответствие правилам структуры (CLAUDE.md):

- `lib/integrations/` — клиент внешней системы с режимами через env, как у DWH.
- `lib/domain/reference/` — чистая логика, без React, Prisma и I/O, тесты рядом.
- `lib/sync/` — оркестрация, единственное место, знающее и про Prisma, и про клиент.
- Существующий `lib/integrations/one-c.ts` **не трогаем**: он про запись заявок
  по Bearer-токену — другая задача, другой способ авторизации.

### Контракт клиента

```ts
export type OneCArticle = {
  uid: string
  code: string | null
  name: string
  parentUid: string | null // null для корня
  isGroup: boolean
  flow: "INFLOW" | "OUTFLOW" | null
  description: string | null
  isDeletedIn1c: boolean
}

export type OneCBankAccount = {
  uid: string
  name: string
  accountNumber: string
  bankName: string
  bankBic: string
  currency: string
  organization: string
  isDeletedIn1c: boolean
}

export type OneCGateway = {
  fetchArticles(kind: ArticleKind): Promise<OneCArticle[]>
  fetchBankAccounts(): Promise<OneCBankAccount[]>
}
```

Реализации: `real` (HTTP) и `fixture` (файл). Выбор — по `ONEC_ODATA_MODE`,
как `DWH_MODE`. Незаданный или неизвестный режим — явная ошибка, не молчаливый
mock (правило уже применено в `one-c.ts`).

### Имена объектов 1С

Точные имена справочников и реквизитов в конфигурации `rbb_cut` неизвестны до
получения прав на OData. Проектное решение: они **не размазываются по коду**, а
собраны в одной константе-карте внутри `one-c-odata.ts` — имя набора OData и
имя каждого поля. Подстановка реальных значений — отдельный шаг плана
реализации (§9), правка одного файла. Ни модель данных, ни логика сравнения,
ни UI от этих имён не зависят.

### Переменные окружения

```
ONEC_ODATA_MODE       # real | fixture
ONEC_ODATA_URL        # http://192.168.79.250:1281/rbb_cut/odata/standard.odata
ONEC_ODATA_USER
ONEC_ODATA_PASSWORD
REFERENCE_SYNC_SECRET # заголовок x-sync-secret для ночного запуска
```

В dev и e2e — `ONEC_ODATA_MODE=fixture`, сеть не задействована.
Добавляются в `.env.example` без значений секретов.

## 5. Модель данных (Prisma)

### Изменения существующих таблиц

К `Article` и `BankAccount` добавляются три поля:

```prisma
externalUid   String?   @unique   // UID записи в 1С; пусто у записей не из 1С
syncedAt      DateTime? @db.Timestamptz(3)
isDeletedIn1c Boolean   @default(false)
```

`isActive`, уже существующий, остаётся механизмом скрытия: пометка удаления в
1С выставляет `isDeletedIn1c = true` и `isActive = false`. Поле изначально
задумывалось под этот сценарий, менять его семантику не требуется.

### Новая таблица журнала

```prisma
enum ReferenceSyncStatus { running  ok  error }
enum ReferenceSyncTrigger { cron  manual }

model ReferenceSyncRun {
  id         String   @id @default(cuid())
  startedAt  DateTime @default(now()) @db.Timestamptz(3)
  finishedAt DateTime? @db.Timestamptz(3)
  status     ReferenceSyncStatus
  trigger    ReferenceSyncTrigger
  created    Int      @default(0)
  updated    Int      @default(0)
  archived   Int      @default(0)
  unchanged  Int      @default(0)
  warnings   Int      @default(0) // напр. нераспознанный тип статьи (§6)
  error      String?

  @@index([status, startedAt])
  @@map("reference_sync_runs")
}
```

**Почему отдельная таблица, а не расширение `SyncRun`.** `SyncRun` обслуживает
синк заявок на оплату со своими счётчиками (`requestsUpserted`, `debitsSkipped`
и др.). Общая таблица означала бы, что в каждой строке половина колонок пуста
по смыслу, а работающий модуль заявок пришлось бы трогать миграцией. Цена
решения — вторая похожая таблица; выгода — работающий модуль остаётся
нетронутым. Осознанный размен в пользу изоляции.

Миграция `onec-reference-sync` — недеструктивная: только новые поля (nullable
либо с дефолтом) и новая таблица.

### Seed

`prisma/seed.ts` создаёт демо-справочники без UID. Их нужно пересоздать с
искусственными, но стабильными `externalUid` (например, `seed-ddc-001`), иначе
первый реальный синк не опознает их как те же записи и создаст дубли. Данные
песочницы расходные, пересоздание безопасно (`/reset-dev`).

## 6. Логика сравнения (`lib/domain/reference/sync-diff.ts`)

Чистая функция: на входе список из 1С и текущее состояние в finflow, на выходе
план изменений. Ни сети, ни Prisma, ни React.

```ts
export type SyncPlan<T> = {
  toCreate: T[]
  toUpdate: { localId: string; data: T }[]
  toArchive: string[] // localId записей, пропавших из 1С или помеченных удалёнными
  unchanged: number
}
```

Правила:

1. Сопоставление — по `externalUid`.
2. Записи, которых нет локально, — в `toCreate`.
3. Записи, у которых **хотя бы одно поле отличается**, — в `toUpdate`.
   Совпадающие целиком не трогаем (иначе каждую ночь переписывался бы весь
   справочник и `syncedAt` терял бы смысл).
4. Локальные записи с `externalUid`, отсутствующие в выгрузке, **или** пришедшие
   с пометкой удаления — в `toArchive`.
5. Локальные записи **без** `externalUid` (остатки ручного ведения) синком не
   трогаются никогда — они ему не принадлежат.
6. **Пустая выгрузка — не повод архивировать.** Если из 1С пришло ноль записей,
   функция возвращает ошибку, а не план «заархивировать всё». Ровно эта защита
   уже стоит в синке заявок (`lib/sync/run-sync.ts`, обработка пустого ответа
   DWH) — принцип переносится.

Маппинг значений 1С в наши enum'ы (`ArticleKind`, `ArticleFlow`) — тоже здесь,
чистыми функциями. Нераспознанное значение типа не роняет синк: пишется `null`,
запись учитывается в счётчике предупреждений запуска.

## 7. Оркестрация (`lib/sync/run-reference-sync.ts`)

Порядок одного запуска:

1. Проверить, что параллельный запуск не идёт (по образцу `run-sync.ts`:
   зависший `running` старше 10 минут не блокирует новый).
2. Создать запись `ReferenceSyncRun` со статусом `running`.
3. Забрать из 1С три справочника: статьи ДДС, статьи БДР, банковские счета.
4. Построить план изменений (§6) для каждого.
5. **Применить всё одной транзакцией Prisma.** Либо приезжает весь снимок, либо
   не меняется ничего — полуобновлённого состояния не бывает.
6. Записать в журнал итог: статус, счётчики, ошибку если была.

### Дерево статей — два прохода

1С указывает родителя по UID, у нас идентификаторы свои. Поэтому внутри
транзакции: сначала записываются все статьи (без связей), затем вторым проходом
проставляется `parentId` по карте `externalUid → localId`. Иначе статья,
приехавшая раньше своей группы, осталась бы без родителя.

Корневой родитель в 1С (нулевой GUID `00000000-…`) отображается в `parentId = null`.

### Поведение при сбоях

| Ситуация | Поведение |
| --- | --- |
| 1С недоступна, таймаут, 401 | запуск → `error`, данные остаются от последней удачной синхронизации |
| Пустая выгрузка | запуск → `error`, ничего не архивируется (§6, п. 6) |
| Ошибка в середине записи | транзакция откатывается целиком |
| Нераспознанный тип статьи | не ошибка: `flow = null`, счётчик предупреждений |

Раздел «Справочники» продолжает работать при любой из этих ситуаций: страницы
читают собственную БД и в 1С при открытии не ходят.

## 8. UI

Источник истины — 1С, поэтому из трёх справочников **удаляются** создание,
редактирование и архивация: server actions и клиентские формы
(`app/reference/*/actions.ts`, формы статей и счетов). Код остаётся в истории
git. Причина удаления, а не скрытия: кнопка, чья правка живёт до ближайшей
ночи, вводит в заблуждение.

Добавляется на страницах справочников:

- подпись «Данные из 1С, обновлено 21 июля, 03:15» (время — `formatDate` из
  `lib/domain/dates.ts`, Москва);
- кнопка **«Обновить из 1С»** — server action, запускает синк вручную,
  показывает результат (сколько создано/обновлено/заархивировано);
- предупреждение, если последний запуск завершился ошибкой: текст ошибки и
  время последнего удачного обновления.

Сохраняется переключатель «показать архивные» — записи, удалённые в 1С, скрыты,
но доступны.

Только shadcn и токены темы; иконки — `lucide-react`. Новых зависимостей нет.

## 9. Проверка реального подключения

Отдельный шаг плана реализации, выполняется после выдачи прав администратором:

1. Запросить корень OData, получить список опубликованных наборов.
2. Найти три нужных справочника, выписать точные имена наборов и реквизитов.
3. Подставить их в карту имён (§4), заполнить фикстуры реальной структурой.
4. Прогнать синк в режиме `real` на dev и сверить выборочно с 1С.

До этого шага вся разработка и приёмка идут на фикстурах.

## 11. Границы

### Входит

Синхронизация трёх справочников из 1С по OData, ночное расписание, ручная
кнопка, журнал запусков, перевод раздела в режим только чтения, тесты.

### Не входит

- Другие справочники: контрагенты, организации, валюты, договоры.
- Запись в 1С из finflow (канал остаётся строго на чтение).
- Мгновенная доставка изменений (push со стороны 1С) — потребовала бы доработок
  на стороне 1С, отдельная задача.
- Привязка статей и счетов к транзакциям и заявкам — по-прежнему отдельная
  фича, `Transaction.category` остаётся текстом.
- Роли и права внутри finflow.

## 12. Открытые вопросы

- Точные имена справочников и реквизитов в конфигурации `rbb_cut` — снимаются
  на шаге §9.
- Есть ли у справочников 1С реквизит даты изменения. Если найдётся — появится
  возможность инкрементального запроса; сейчас архитектура его не требует и от
  ответа не зависит.
- Что делать, если в 1С заархивирована группа с активными потомками. Текущее
  поведение: архивируется только сам узел, потомки остаются (наследуется из
  июльской спеки). Уточнить на реальных данных.
