# Спека: светофор авто-проверок и контекст карточки заявки

Дата: 2026-07-14
Статус: дизайн утверждён (брейншторминг с Минасом), готово к плану реализации.
Источники требований: `ТЗ_Карточка_заявки.docx` (репо `fin`, включая дополнения v1.1
со встречи 18.05.2026), рабочая реализация светофора (`fin/composables/useVerdict.ts`,
`fin/finflow-app/frontend/src/reports/requestVerdict.js`), HTML-макеты карточки и
сводного экрана (репо `fin`). Строится поверх утверждённой спеки
`2026-07-14-payment-requests-design.md` (далее — «спека заявок»).

## 1. Контекст и проблема

Спека заявок закрывает контроль исполнения, но собственник по-прежнему принимает
решение о согласовании вслепую: не видно, хватает ли денег, в плюсе ли фонд,
новый ли контрагент, есть ли основание. В старом приложении (репо `fin`) для этого
был «светофор» — автоматическая оценка заявки перед согласованием + контекстные
секции в карточке. Этот модуль переносит светофор в finflow, исправляя известные
недочёты старой реализации.

Цель (из ТЗ): собственник за 5–10 секунд видит — достаточно ли денег, в плюсе ли
фонд, нормальный ли контрагент, есть ли основание — и принимает решение без
переключения между отчётами.

## 2. Границы

### Входит

1. **Светофор**: 5 авто-проверок, совокупный вердикт 🟢/🟡/🔴, чек-лист в карточке.
2. **Контекст карточки заявки**: секции «Ликвидность», «Фонд», «Контрагент»,
   «Заказ / Основание», «Вложения» (метаданные), «Связанные заявки», шапка
   с чипами и руководителем отдела.
3. **Сводка в реестре** (решение: один экран, реестр обрастает сводкой):
   полоса метрик, сворачиваемая панель остатков по счетам и фондов, колонка
   вердикта, массовое согласование только 🟢.
4. **Страница настроек светофора**: пороги + флаг «учитывать в вердикте».
5. Синк новых срезов данных через per-slice гейтвеи (API 1С / DWH / fixture).

### Не входит (осознанно)

- **Проверки «Соответствие финплану» и «Заранее согласовано»** — источник
  (финмодель Антонины) существует только в Excel вне 1С; добавятся, когда
  финмодель приедет в DWH. В типах и настройках проверки заведены заранее,
  в UI отображаются серым «нет данных» вне вердикта.
- **Интеграция с финмоделью** (привязка платежей к неделям, вкладка
  «Согласованные платежи», авто-корректировка плана) и **прогноз кассовых
  разрывов** (ТЗ §10.4–10.6) — отдельные темы, добавлены в бэклог.
- **Скачивание файлов вложений** — в v1 только метаданные; файлы — когда
  появится метод API 1С (кандидат в бэклог).
- **Кнопка «Запросить уточнение» у инициатора** (ТЗ §3.11, §6.2) — требует
  канала уведомлений, которого нет; кандидат в бэклог вместе с уведомлениями.
- **Excel-экспорт реестра** (ТЗ §10.3) — кандидат в бэклог.
- Telegram WebApp / мобильная версия для бота — вне текущего приложения.

## 3. Ключевые решения

| Тема | Решение |
|---|---|
| Компоновка сводки | Один экран: `app/requests` получает метрики и сворачиваемую панель остатков/фондов (не отдельный dashboard) |
| Состав проверок v1 | 5 надёжных: деньги, фонд, основание, заказ/договор, история контрагента |
| Уровни вердикта | Три автоматических (🟢 ok / 🟡 warn / 🔴 bad); ⚫ block зарезервирован в типах, правило — позже по реальной статистике |
| Основание платежа | Заказ поставщику — главная сущность (ТЗ §10.2), договор справочно; услуги без заказа проверяются по договору |
| Вложения | Только метаданные (имя, тип, дата) из DWH; файлы позже |
| Массовое согласование | Только заявки с вердиктом 🟢; у 🟡/🔴 чекбокс неактивен, подсказка «только через карточку» |
| Пороги и флаги | В БД, редактируются на странице настроек (по образцу статей ДДС) |
| Источники срезов | Гибрид: интерфейс на срез; реализация 1С API / DWH / fixture выбирается конфигурацией per slice |
| Вердикт | Не материализуется: чистая функция при чтении из локальных срезов |
| Баг старого кода | Проверка фонда считает остаток **после** платежа (старый код не вычитал сумму заявки) |

## 4. Архитектура

Принцип спеки заявок сохраняется: UI читает только свой PostgreSQL; внешние
системы трогает фоновый синк.

```
ИСТОЧНИКИ                ГЕЙТВЕИ (lib/integrations/)          СИНК             ЧТЕНИЕ
─────────                ───────────────────────────          ────             ──────
REST API 1С ──────┬────► balances    (get/balance)    ─┐
(Bearer, доступен ├────► funds       (get/fund)       ─┤
 уже сейчас)      └────► rates       (get/currencyRate)┤
                                                        ├──► lib/sync/ ──► PostgreSQL
DWH DEEONE ───────┬────► partnerHistory (вьюха, заказать)┤    (upsert         │
(доступ обещан,   ├────► supplierOrders (вьюха, заказать)┤     срезов)        ▼
 не подтверждён)  └────► attachmentsMeta(вьюха, заказать)┘              server components
                                                                              │
Fixtures (json) ──────► любой срез до готовности источника                    ▼
                                                             lib/domain/requests/verdict.ts
                                                             чистая функция, 5 проверок,
                                                             пороги из БД, при рендере
```

- **Интерфейс на срез, не на источник**: `BalancesGateway`, `FundsGateway`,
  `RatesGateway`, `PartnerHistoryGateway`, `SupplierOrdersGateway`,
  `AttachmentsMetaGateway`. Синку всё равно, откуда данные.
- **Источник каждого среза — конфигурация**: `SLICE_BALANCES_SOURCE=1c|dwh|fixture`
  и аналогичные. Доступ к базе обещан, но не подтверждён — когда прояснится,
  переключается env-переменная, код не меняется.
- **Fixture-реализация у каждого гейтвея** — весь модуль разрабатывается
  и демонстрируется без внешних систем (паттерн плана 03).
- Чтение остатков/фондов/курсов через REST API 1С — осознанное расширение
  правила «API 1С только запись» из спеки заявок: у API уже есть готовые
  методы `get/balance`, `get/fund`, `get/currencyRate`, что снимает
  зависимость 3 из 5 проверок от домашки по DWH.

## 5. Доменная логика вердикта

`lib/domain/requests/verdict.ts` — чистая функция без I/O, unit-тесты рядом:
`computeVerdict(request, slices, settings) → { level, title, description, checks[] }`.

Каждая проверка возвращает `{ id, label, status: ok|warn|bad|info, sublabel,
excludedFromVerdict }`. Вердикт = худший статус проверок, у которых
`includeInVerdict = true` (настройка) и статус не `info`.

| # | Проверка | 🟢 ok | 🟡 warn | 🔴 bad |
|---|---|---|---|---|
| 1 | Деньги на счёте | Остаток счёта списания ≥ суммы | На счёте не хватает, но по всем счетам юрлица (₽ экв. по курсам) хватает — «нужен перевод» | Не хватает по юрлицу целиком |
| 2 | Остаток фонда | Остаток фонда **после платежа** ≥ 0 | Уходит в минус ≤ `fundDeficitPercent` (20%) от плана недели | Глубже порога |
| 3 | Документ-основание | Есть вложения (по метаданным) | Только текстовый комментарий | Ничего |
| 4 | Заказ / договор | Есть заказ, оплачено с этим платежом ≤ 100%; либо заказа нет, но договор активен и платёж в рамках суммы | Переплата по заказу > 100%; либо платёж превысит сумму договора | Ни заказа, ни договора; договор закрыт |
| 5 | История контрагента | Постоянный: ≥ `minOperationsForConstant` (3) платежей | Эпизодический (1–2) или пауза > `oldPartnerMonths` (12 мес) | Первый платёж |

Правила:

- **Деградация**: срез недоступен/пуст → проверка `info` «нет данных»,
  из вердикта исключается автоматически; светофор сужается до проверок
  с данными, но не падает.
- **Мультивалютность**: суммы приводятся к ₽ по срезу курсов.
- **Уровни**: 🟢 «Можно согласовать», 🟡 «Можно согласовать с оговоркой»,
  🔴 «Требует внимания». ⚫ «Заблокировано» в enum зарезервирован, автоматически
  не выставляется.
- Проверки «финплан» (`finplan`) и «заранее согласовано» (`preapproved`)
  заведены в реестре проверок, всегда `info` до появления источника.

## 6. Модель данных (Prisma)

Стиль спеки заявок: деньги — BigInt в копейках, даты — `Timestamptz(3)`,
upsert по UID из 1С.

Новые таблицы-срезы:

- **`AccountBalance`** — `accountUid @unique`, `orgUid`, `orgName`, `accountName`,
  `bankName`, `currency`, `balanceMinor BigInt`, `syncedAt`.
- **`FundSnapshot`** — `fundUid @unique`, `name`, `planWeekMinor`, `factWeekMinor`,
  `balanceMinor`, `syncedAt`.
- **`CurrencyRate`** — `currencyCode @unique`, `rate Decimal`, `rateDate`, `syncedAt`.
- **`PartnerStats`** — `partnerUid @unique`, `firstOperationAt`, `paymentCount`,
  `totalPaidMinor`, `receivableMinor` (дебиторка), `payableMinor` (кредиторка),
  `recentPayments Json` (последние 3–5: дата · основание · сумма), `chatUrl`
  (ссылка на чат из карточки контрагента — пригодится отправке платёжек), `syncedAt`.
- **`PartnerContract`** — `contractUid @unique`, `partnerUid`, `number`, `date`,
  `isActive`, `amountMinor`, `paidMinor`, `debtMinor` (задолженность по договору,
  ТЗ §10.1), `currency`, `syncedAt`.
- **`SupplierOrder`** — `orderUid @unique`, `partnerUid`, `contractUid?`, `number`,
  `date`, `amountMinor`, `paidMinor`, `currency`, `syncedAt` — источник процента
  оплаты по заказу.
- **`AttachmentMeta`** — `requestUid`, `fileName`, `fileType`, `createdAt`,
  `@@unique([requestUid, fileName])`.

Расширение **`PaymentRequest`** (новые поля из выгрузки заявок):
`debitAccountUid` (счёт списания — нужен проверке №1), `contractUid`, `orderUid`,
`initiatorHead` (руководитель отдела, ТЗ §10.1). Все nullable — старые данные
не мигрируются.

Настройки светофора:

- **`VerdictThreshold`** — `key @unique`, `value Decimal`. Сид: `fundDeficitPercent=20`,
  `oldPartnerMonths=12`, `minOperationsForConstant=3`. Переплата по заказу
  порога не имеет: любое превышение 100% — 🟡.
- **`VerdictCheckSetting`** — `checkId @unique`, `includeInVerdict Boolean`.
  Сид: все 5 активных проверок `true`, `finplan`/`preapproved` — `false`.

**Связанные заявки** — не таблица: запрос по `PaymentRequest` (тот же `partnerUid`
или `orderUid`, `payDate` ± 30 дней).

`SyncRun` расширяется JSON-полем со счётчиками по срезам
(`slices: { balances: 12, funds: 3, ... }`).

## 7. Экраны

По образцу существующих: server components + Prisma + `lib/domain/` + shadcn +
server actions.

### 7.1 Реестр `app/requests` (расширение)

- **Полоса метрик**: на согласовании (всего · срочных), сумма к оплате на 7 дней
  (₽ экв.), красные флаги (кол-во · сумма; клик = фильтр «только с проблемами»),
  остаток группы ₽.
- **Сворачиваемая панель «Остатки и фонды»**: таблица счетов по юрлицам и валютам
  с колонками «до / после отмеченных» (проекция пересчитывается на клиенте от
  отмеченных чекбоксов); карточки фондов (план недели · факт · остаток),
  клик по фонду = фильтр реестра.
- **Таблица**: колонка вердикта (цветная точка + подпись); чипы (срочно, новый
  поставщик, фонд в минусе); чекбоксы активны только у 🟢 (у прочих disabled
  с подсказкой «только через карточку»); «Согласовать выбранные» — как в спеке
  заявок, но с ограничением по вердикту.
- **Фильтры**: к существующим добавляются «контрагент» и «только с проблемами».
- Вердикты для реестра вычисляются на сервере одной выборкой срезов
  (заявок на согласовании — десятки, это дёшево).

### 7.2 Карточка `app/requests/[uid]` (расширение)

Двухколоночная. Слева — секции, каждая несёт индикатор «своей» проверки:

1. **Шапка**: № · даты · заголовок; организация · отдел · инициатор ·
   руководитель отдела; сумма в валюте + ₽ экв. с курсом; чипы (срочно,
   аванс N%, тип контрагента).
2. **Ликвидность**: счёт списания подсвечен (остаток до/после), итоги по
   юрлицу и группе; при нехватке — баннер с рекомендацией (перевод/конвертация).
3. **Фонд**: план недели · факт · остаток, прогресс-бар, «заявка изменит
   остаток на X ₽».
4. **Контрагент**: тип отношений, дата первой операции, кол-во платежей,
   общая сумма, дебиторка/кредиторка, кнопка 💬 (если `chatUrl` есть),
   последние 3–5 платежей.
5. **Заказ / Основание**: договор справочно (№, дата, статус, задолженность);
   заказ — плитки «сумма · оплачено ранее (%) · с этим платежом (%)»,
   прогресс-бар (зелёный ≤ 100%, красный > 100%), будущие заявки по заказу.
6. **Вложения**: список метаданных (имя, тип, дата), без скачивания.
7. **Комментарий инициатора**.
8. **Исполнение** — блок из спеки заявок, без изменений.
9. **Связанные заявки ± 30 дней**: №, описание, сумма, статус.

Справа — **панель авто-проверки**: вердикт (цвет + заголовок + описание),
чек-лист всех проверок (включая серые «нет данных»), `syncedAt` худшего
из задействованных срезов; ниже — кнопки «Согласовать» / «Отклонить
с причиной» из спеки заявок.

### 7.3 Настройки `app/settings/verdict`

Таблица порогов (`VerdictThreshold`) и флагов «учитывать в вердикте»
(`VerdictCheckSetting`), по образцу `app/settings/cash-flow-items`.
Изменения действуют немедленно (вердикт не материализован).

## 8. Синхронизация, ошибки, деградация

- Синк срезов — шаги существующего пайплайна (`POST /api/jobs/sync`, то же
  расписание, тот же `SyncRun`). Упавший срез не валит синк: остальные шаги
  выполняются, ошибка шага пишется в `SyncRun`, срез остаётся на прошлых данных.
- Источник каждого среза — env-конфигурация (`1c | dwh | fixture`).
- Пустой/незаказанный срез → соответствующая проверка `info`, секция карточки
  показывает причину и время последнего успешного обновления.
- Таймауты на все внешние вызовы; ретраев внутри запуска нет — подберёт
  следующий запуск по расписанию.
- Секреты (Bearer API 1С, строка DWH) — только env, в git не попадают.

## 9. Тестирование

- **Unit (vitest, `lib/domain/`)**: `verdict.ts` — все проверки по таблице §5,
  границы порогов (ровно 20% фонда, ровно 100% заказа, ровно 3 платежа),
  мультивалютный пересчёт, деградация «нет данных», флаги `includeInVerdict`,
  сборка худшего статуса; расчёт «после платежа» для фонда и счёта.
- **Интеграционные**: синк срезов на замоканных гейтвеях — upsert,
  идемпотентность, частичный успех, переключение источника конфигурацией.
- **E2e (Playwright)**: seed-смоук — реестр с вердиктами и disabled-чекбоксами,
  панель остатков с проекцией «после отмеченных», карточка с чек-листом и
  серыми проверками, изменение порога в настройках меняет вердикт.
- **Seed**: заявки на все три вердикта + случаи деградации (без контрагента,
  без заказа, без вложений, пустой срез фондов).

## 10. Этапность реализации (укрупнённо, детали — в плане)

1. Prisma-модели срезов + `verdict.ts` + fixture-гейтвеи + seed —
   светофор работает целиком без внешних систем.
2. Карточка: секции контекста + панель авто-проверки.
3. Реестр: колонка вердикта, метрики, панель остатков/фондов,
   массовое согласование только 🟢.
4. Страница настроек порогов.
5. Боевые гейтвеи: сначала API 1С (`get/balance`, `get/fund`,
   `get/currencyRate` — доступны уже сейчас), затем DWH-вьюхи по мере
   готовности (заказ — §11).

## 11. Предпосылки и открытые вопросы

1. **Вьюхи DWH** (заказать владельцу DWH, тем же пакетом, что и вьюхи спеки
   заявок): взаиморасчёты с контрагентами (+ последние платежи), договоры
   с задолженностью, заказы поставщику с суммой оплат, метаданные вложений.
   *(До готовности — fixtures; проверки 3–5 в деградации на реальных данных.)*
2. **Поля в выгрузке заявок**: подтвердить наличие счёта списания, договора,
   заказа, руководителя отдела. *(Влияет на проверки №1 и №4 и шапку карточки.)*
3. **Формат методов API 1С** `get/balance`, `get/fund`, `get/currencyRate` —
   есть только названия, схемы ответов нет. Проверить и зафиксировать.
4. **План недели по фондам**: уточнить у Антонины, живёт ли он в 1С
   («Распределение по фондам») или только в финмодели-Excel. Если только
   в Excel — проверка №2 стартует в деградации (остаток фонда без плана недели:
   ok/bad без жёлтой зоны).
5. **Ссылка на чат** в карточке контрагента 1С → доставка в DWH (общая
   предпосылка со спекой заявок §11.4).

## 12. Пополнение бэклога (по итогам этого дизайна)

- Интеграция с финмоделью: привязка платежей к неделям, вкладка
  «Согласованные платежи», авто-корректировка плана (ТЗ §10.4–10.5).
- Прогноз кассовых разрывов (ТЗ §10.6).
- Проверки «финплан» и «заранее согласовано» — активация после доставки
  финмодели в DWH.
