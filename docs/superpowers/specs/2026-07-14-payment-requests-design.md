# Спека: модуль «Заявки на оплату» — контроль исполнения и отправка платёжек

Дата: 2026-07-14
Статус: дизайн утверждён (брейншторминг с Минасом), готово к плану реализации.
Источник требований: транскрибация аналитика (бухгалтера) + исследование старого
приложения (репозиторий `fin`: Nuxt → React+Flask, `ТЗ_Карточка_заявки.docx`).

## 1. Контекст и проблема

В группе компаний работает процесс оплаты счетов поставщиков:

1. Сотрудники присылают бухгалтеру пожелания на оплату счетов.
2. Бухгалтер формирует в 1С документ **«Заявка на расход денежных средств»**
   (кому, сколько, за что, по какому договору).
3. Заявка уходит на согласование руководителю компании (сейчас — через мессенджер).
4. Согласованная заявка идёт в оплату.
5. На следующий рабочий день до 11:00 в 1С автоматически загружается **выписка
   банка**; списание фиксируется документом **«Расход ДС» (по банку/кассе)**
   с привязкой к заявке-основанию. Исключение: Азия-банк — выписка автоматически
   не грузится (причина выясняется с банком).
6. По оплатам «за товар» бухгалтер вручную отправляет **платёжное поручение
   с отметкой банка** в чат поставщика в Яндекс Мессенджере (ссылка на чат
   есть в карточке контрагента в 1С).

Боли, которые закрывает модуль:

- **Не видно, исполнена ли согласованная заявка.** Нужен контроль: списание
  произошло — заявка «зелёная»; списания нет в срок — «красная», и тогда
  бухгалтер либо переносит дату оплаты (в 1С), либо объясняет причину.
- **Отправка платёжек поставщикам — ручная рутина.** Нужно автоматизировать
  подбор платёжки и чата с подтверждением бухгалтером перед отправкой.

Старое приложение (репо `fin`) переписываем: этот модуль строится с нуля
в приложении **finflow** (этот репозиторий).

## 2. Границы

### Входит

1. **Реестр заявок и согласование**: сводный экран заявок, карточка заявки,
   действия «Согласовать» / «Отклонить с причиной».
2. **Контроль исполнения**: статусы исполнения (зелёная/красная), дедлайн
   по выписке, комментарии бухгалтера к красным заявкам.
3. **Отправка платёжек**: очередь на подтверждение, отправка файла в чат
   поставщика в Яндекс Мессенджере, журнал отправок.
4. Настройка статей ДДС («оплата за товар») и журнал синхронизаций.

### Не входит (осознанно, следующие итерации)

- **Светофор авто-проверок из 7 пунктов** (остатки на счетах, фонды, финплан,
  история контрагента, договор — см. `ТЗ_Карточка_заявки.docx` в репо `fin`).
  Требует ещё 4–5 срезов данных из DWH — отдельная спека. Карточка заявки в v1
  показывает реквизиты заявки и блок исполнения.
- **Авторизация и роли в приложении.** По спеке окружения
  (`2026-07-13-team-environment-design.md`) приложение закрыто nginx basic auth,
  внутренняя авторизация — отдельная будущая фича. Ролевая модель здесь описана
  концептуально и станет требованием к той фиче.
- Отчёты ИУ/БДР, ИИ-чат и прочая функциональность старого приложения.

## 3. Роли (концептуально, без реализации прав в v1)

| Роль | Что делает в модуле |
|---|---|
| Собственник / руководитель | Смотрит реестр и карточки, согласует/отклоняет, видит красные заявки и объяснения бухгалтера |
| **Бухгалтер** (новая роль) | Контролирует исполнение, комментирует красные, подтверждает отправку платёжек, управляет статьями ДДС |
| Читатель | Только просмотр |

В v1 все действия доступны каждому, кто прошёл basic auth. Когда появится
авторизация — права раскладываются по этой таблице.

## 4. Архитектура

Принцип: **UI никогда не ходит во внешние системы синхронно при открытии
страницы.** Чтение — только из своей PostgreSQL; внешние системы трогают
фоновый синк и явные действия пользователя.

```
                 ┌──────────────────────────── finflow (Next.js) ───────────────────────────┐
DWH DEEONE       │  lib/integrations/dwh.ts ──► lib/sync/ ──► PostgreSQL (Prisma)           │
(SQL Server) ◄───┼── SSH-туннель host02 → ai01        ▲                │                    │
                 │                                    │                ▼                    │
cron / кнопка ───┼──► POST /api/jobs/sync ────────────┘   app/requests, app/dispatch, ...   │
                 │                                                     │                    │
REST API 1С ◄────┼── server actions «Согласовать/Отклонить» ◄──────────┤                    │
(Bearer)         │                                                     │                    │
Яндекс Мессенджер◄── lib/integrations/yandex-messenger.ts ◄─ подтверждение бухгалтера       │
                 └──────────────────────────────────────────────────────────────────────────┘
```

Разделение источников (решение утверждено):

- **Всё чтение — из DWH/DEEONE** (SQL Server, доступ по SSH-цепочке
  host02 → ai01; канал уже используется старым приложением, см. `fin/finflow-app/dwh.py`).
- **REST API 1С (Bearer, `http://192.168.79.250:4480/api/1crm/...`) — только
  запись**: `POST /post/approveBid`, `POST /post/declineBid`.
- **Яндекс Мессенджер Bot API** — доставка платёжек в чаты поставщиков.
- Файлы платёжек — папка на сервере (процесс выкладки пока не настроен, см. §11).

Новая npm-зависимость: **`mssql`** (драйвер SQL Server) — одобрена Минасом
в рамках этого дизайна. SSH-туннель поднимается системно (systemd/autossh),
приложение подключается к локальному порту.

## 5. Модель данных (Prisma)

Все даты — `@db.Timestamptz(3)`. Суммы заявок и списаний — **BigInt в копейках**
(`Int` ограничен ±21,4 млн ₽ — заявки за товар бывают больше).

- **`PaymentRequest`** — снапшот заявки из DWH:
  `uid` (UID 1С, `@unique`), `number`, `date`, `orgName/orgInn/orgUid` (юрлицо),
  `initiator`, `department`, `amountMinor BigInt`, `currency`,
  `cashFlowItem` (статья ДДС), `fund`, `partnerName/partnerInn/partnerUid`,
  `payDate` (плановая дата оплаты), `approvalStatus` (из 1С: on_approval /
  approved / declined), `importance`, `comment`,
  `executionStatus` (вычисляется, см. §6), `executedAt`,
  `isDeletedIn1c Boolean` (пропала из выгрузки — помечаем, не удаляем),
  `syncedAt`.
- **`Debit`** — списание из выписки (документ «Расход ДС»):
  `docUid @unique`, `date`, `amountMinor BigInt`, `bankAccount`, `bankName`,
  `requestUid` (привязка к заявке из регистра 1С), `syncedAt`.
- **`ExecutionComment`** — объяснение бухгалтера к красной заявке:
  `requestId`, `text`, `author` (строка в v1 — авторизации нет), `createdAt`.
  Живёт только в приложении (гибридное решение: дату оплаты переносят в 1С,
  объяснение пишут здесь).
- **`PaymentOrderDispatch`** — отправка платёжки:
  `requestId`, `debitId`, `fileName/filePath` (nullable — файл мог не найтись),
  `chatId/chatUrl` (nullable), `status` (awaiting_confirmation / not_ready /
  sent / failed / skipped), `confirmedBy`, `sentAt`, `error`, `skipReason`.
  `@@unique([requestId, debitId])` — защита от двойной отправки.
- **`CashFlowItemSetting`** — статьи ДДС: `name @unique`, `isGoods Boolean`
  (флаг «оплата за товар» → триггер отправки платёжки).
- **`SyncRun`** — журнал синков: `startedAt`, `finishedAt`, `status`
  (running / ok / error), `requestsUpserted`, `debitsUpserted`, `error`,
  `trigger` (cron / manual). Питает индикатор свежести данных.

## 6. Бизнес-логика статусов исполнения

Чистая функция в `lib/domain/requests/execution-status.ts` (без I/O, unit-тесты
рядом): `computeExecutionStatus(request, debits, now) → status`.

| Статус | Условие | Отображение |
|---|---|---|
| `on_approval` | не согласована и не отклонена | серый |
| `declined` | отклонена | серый, приглушённый |
| `awaiting` | согласована, дедлайн выписки не наступил | нейтральный |
| `executed` | есть ≥1 привязанное списание | **зелёный** |
| `overdue` | согласована, списаний нет, дедлайн прошёл | **красный** |

**Дедлайн** = следующий рабочий день после `payDate`, **11:00 по Москве**
(к этому времени выписка за день оплаты гарантированно загружена в 1С).
Рабочие дни v1: пропускаем сб/вс; производственный календарь праздников — §11.

Правила:

- Перенос даты оплаты в 1С → при следующем синке `payDate` обновится и заявка
  вернётся из красной в `awaiting` автоматически.
- Красная заявка с комментарием бухгалтера остаётся красной, но в UI помечена
  «есть объяснение» — руководитель видит и просрочку, и причину.
- Исполнение = наличие привязанного списания. Частичные оплаты v1 не
  интерпретирует (показываем сумму списаний рядом с суммой заявки); правило
  «исполнена частично» — открытый вопрос §11.
- Авторитетный статус — хранимый `executionStatus`: пересчитывается при каждом
  синке (единая точка истины для отображения и фильтров). В критичное окно
  вокруг дедлайна 11:00 синк идёт каждые 15 минут, так что задержка перехода
  в «красную» не превышает 15 минут — приемлемо.

## 7. Синхронизация с DWH

- Логика: `lib/integrations/dwh.ts` (подключение, SQL) + `lib/sync/`
  (оркестрация: upsert заявок, списаний, пересчёт статусов, создание
  черновиков отправок §8).
- Единая точка запуска — route handler **`POST /api/jobs/sync`**.
  Авторизация: заголовок с cron-секретом (env) — для планировщика; либо
  запрос из сессии (basic auth) — для кнопки «Обновить» в UI.
- Расписание (системный cron на сервере): **9:00–12:00 МСК каждые 15 минут**
  (окно загрузки выписок), **затем раз в час до 19:00 МСК**; ночью и в
  выходные синк не запускается.
- Синк идемпотентный: upsert по `uid`/`docUid`; заявки, исчезнувшие из
  выгрузки, помечаются `isDeletedIn1c`, не удаляются.
- Каждый запуск пишет `SyncRun`; конкурентные запуски исключаются
  (advisory lock / проверка `running`).
- Глубина выборки: заявки и списания за скользящее окно (по умолчанию 90 дней,
  env-настройка) + все незакрытые (не исполненные) заявки старше окна.

## 8. Отправка платёжек поставщикам

Триггер (утверждено): **авто-подбор + подтверждение бухгалтера**. Критерий
«оплата за товар» — статья ДДС заявки входит в настраиваемый список
(`CashFlowItemSetting.isGoods`).

Пайплайн:

1. Синк обнаружил новое списание по заявке со статьёй «за товар» → создаётся
   `PaymentOrderDispatch`. Приложение пытается подобрать:
   - **файл платёжки** из папки на сервере (правило сопоставления — по данным
     из имени/содержимого файла: номер платёжки, дата, сумма; уточняется, §11);
   - **чат поставщика** — из поля карточки контрагента 1С (должно приезжать
     через DWH, §11).
2. Всё нашлось → статус `awaiting_confirmation`; чего-то нет → `not_ready`
   с перечнем недостающего; бухгалтер может прикрепить файл и указать чат
   вручную (fallback v1, работает и до готовности предпосылок).
3. Экран `app/dispatch/`: очередь (заявка, поставщик, сумма, файл со
   скачиванием, чат-получатель), действия «Отправить», «Отправить все
   готовые», «Пропустить» (с причиной).
4. Подтверждение → `lib/integrations/yandex-messenger.ts` загружает файл
   и шлёт сообщение в чат (Bot API Яндекс 360). Успех → `sent` (кто, когда);
   ошибка → `failed` с текстом и кнопкой «Повторить». Автоматических ретраев
   нет — защита от спама в чат поставщика.

## 9. Экраны

Все — по образцу `app/transactions/`: server component + Prisma + доменная
логика из `lib/domain/` + shadcn + server actions + `useActionState`.

1. **`app/requests/`** — реестр: таблица заявок с цветом статуса исполнения,
   фильтры (статус, юрлицо, фонд, период, «только красные»), строка свежести
   («Данные на 10:42» из `SyncRun`) + кнопка «Обновить», мультивыбор →
   «Согласовать выбранные».
2. **`app/requests/[uid]/`** — карточка: реквизиты заявки, блок «Исполнение»
   (статус, дедлайн, привязанные списания, история), комментарии бухгалтера
   (форма), действия «Согласовать» / «Отклонить с причиной».
3. **`app/dispatch/`** — очередь и журнал отправок платёжек (§8).
4. **`app/settings/cash-flow-items/`** — статьи ДДС с флагом «оплата за товар»
   (список статей пополняется синком, флаг ставится здесь).

Согласование: server action → `POST /api/1crm/post/approveBid` /
`declineBid` (Bearer из env `API_TOKEN`). Успех → обновляем `approvalStatus`
в своей БД (до подтверждения следующим синком) + `revalidatePath`. Ошибка/
недоступность 1С → `{ error }` в форму, состояние не меняется.

## 10. Обработка ошибок и надёжность

- DWH недоступен → `SyncRun.status = error`; UI показывает баннер «данные
  устарели, последнее обновление N», работает на последних данных.
- 1С API недоступен → ошибка в форме согласования, ничего не меняется.
- Ошибка отправки в ЯМ → `failed` + текст, повтор только вручную.
- Дубли отправок исключены уникальным индексом `[requestId, debitId]`.
- Все внешние вызовы с таймаутами; секреты (строка подключения DWH,
  `API_TOKEN`, токен бота, cron-секрет) — только в env, в git не попадают.
- Деньги: `BigInt` копейки, форматирование в `lib/domain/money` (без float).

## 11. Предпосылки и открытые вопросы

Блокеры отдельных частей (не всего модуля) — закрываются параллельно
с разработкой; до закрытия работают fallback-режимы:

1. **Схема DWH**: какие таблицы/вьюхи DEEONE содержат заявки, списания
   («Расход ДС») и привязку списание→заявка. Предпочтительно заказать
   у владельца DWH готовые вьюхи — это чистый контракт. *(Блокирует синк.)*
2. **Файлы платёжек**: процесс выкладки не настроен — кто выкладывает,
   в какую папку, формат имён (от этого зависит правило сопоставления).
   *(Fallback: ручное прикрепление файла бухгалтером.)*
3. **Бот Яндекс Мессенджера**: создать бота (Яндекс 360), получить токен,
   добавить в чаты поставщиков. *(Fallback: ручной выбор чата, статус not_ready.)*
4. **Ссылка на чат в карточке контрагента 1С**: подтвердить поле и его
   доставку в DWH.
5. **Азия-банк**: выписка не грузится автоматически → заявки по счетам
   Азия-банка будут ложно краснеть. Аналитик выясняет причину с банком;
   до решения бухгалтер закрывает их комментарием.
6. **Производственный календарь**: v1 считает рабочими все дни кроме сб/вс;
   праздники дают ложные красные утром — при необходимости добавим календарь.
7. **Частичные оплаты**: правило статуса при списании меньше суммы заявки.
8. **Статусы согласования в данных DWH**: как кодируется `statusNow`
   (значения) — сверить при получении доступа к данным.

## 12. Тестирование

- **Unit (vitest, рядом с кодом, `lib/domain/`)**: `execution-status`
  (дедлайн 11:00 МСК, выходные, перенос даты, отклонённые, граница
  «до/после 11:00»), критерий «за товар», сопоставление файла платёжки,
  деньги BigInt.
- **Интеграционные**: `lib/sync/` на замоканном ответе DWH (upsert,
  идемпотентность, пометка удалённых, создание черновиков отправок);
  server actions согласования на замоканном 1С (успех/ошибка/таймаут).
- **E2e (Playwright, `tests/e2e/`)**: смоук на seed-данных — реестр с цветами
  и фильтром «красные», карточка с комментарием, очередь отправки
  с мок-мессенджером (отправка/ошибка/повтор).
- **Seed**: демо-заявки во всех статусах, списания, статьи ДДС с флагами,
  отправки в разных статусах.

## 13. Этапность реализации (укрупнённо, детали — в плане)

1. Схема Prisma + доменная логика статусов + seed + реестр и карточка
   на seed-данных (работает без внешних систем).
2. Синк из DWH (после закрытия вопроса №1 §11) + индикатор свежести.
3. Согласование через REST API 1С.
4. Комментарии бухгалтера.
5. Отправка платёжек: очередь и журнал на ручном режиме → авто-подбор файла
   и чата → интеграция с ботом ЯМ (по мере закрытия предпосылок №2–4 §11).
