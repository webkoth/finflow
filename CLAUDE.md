# finflow — правила работы

Внутреннее финансовое приложение. Его развивают доменные специалисты (не разработчики)
через Claude Code. Эти правила обязательны для каждой сессии. Полная спецификация
окружения: `docs/superpowers/specs/2026-07-13-team-environment-design.md`.

**Терминология:** «песочница» в речи специалистов = dev-окружение (ветка `develop`,
dev-контур). Это синонимы: если пользователь говорит «выложи в песочницу» —
это доставка в dev через `/ship`. В коде, ветках, конфигах и документах
используется только техническое имя dev/develop.

## Жёсткие рамки

- Стек только такой: Next.js (App Router) + TypeScript + Tailwind + shadcn/ui +
  Prisma + PostgreSQL. Никаких других фреймворков, сборщиков, ORM и CSS-подходов.
- **Новые npm-зависимости — только с одобрения разработчика (Минас).** Если для
  задачи кажется нужной новая библиотека — остановись и спроси, не устанавливай.
- Ветка `main` — production, в неё нельзя пушить и мержить локально. Путь в prod
  один: PR `develop` → `main`, ревью разработчика.

## Процесс разработки

- Любая новая функциональность или изменение поведения начинается со скилла
  **brainstorming** (спека → план → реализация). Не пиши код фичи без утверждённого
  дизайна — даже «простой».
- Мелкие правки (опечатка, текст на кнопке) можно делать без полного цикла,
  но через обычные проверки перед коммитом.
- Образцовая фича — `app/transactions/`: server component (страница) + Prisma +
  доменная логика из `lib/domain/` + shadcn + server action + клиентская форма
  через `useActionState`. Новые фичи строй по этому образцу.

## Структура каталогов

- `app/<раздел>/` — страницы, layout'ы и server actions раздела.
- `components/ui/` — только компоненты shadcn (не редактировать руками).
- `components/` (вне ui/) — переиспользуемые композиции из shadcn-примитивов.
- `lib/domain/` — чистая доменная логика (расчёты, деньги, даты, агрегации).
  Без React, без Prisma, без I/O. Обязательные unit-тесты рядом с кодом.
- `lib/db.ts` — единственная точка доступа к Prisma-клиенту в коде приложения
  (standalone-скрипты вроде `prisma/seed.ts` — допустимое исключение).
- `lib/integrations/` — клиенты внешних систем (DWH, 1С); режимы mock/fixture
  выбираются через env, без сети в dev/e2e.
- `lib/sync/` — оркестрация синка данных из внешних систем в свою БД.
- `prisma/` — схема, миграции, seed.
- `tests/e2e/` — Playwright-смоук.

## Мутации данных (паттерн)

- Мутации — только через server actions с сигнатурой
  `(prevState: FormState, formData: FormData) => Promise<FormState>`.
- Ожидаемые ошибки (валидация) НЕ бросаются — возвращаются как `{ error: string }`,
  форма показывает их через `useActionState` (см. `app/transactions/transaction-form.tsx`).
- Неожиданные сбои ловит глобальный `app/error.tsx` (в этой версии Next используется
  `unstable_retry`, не `reset`).
- После успешной мутации — `revalidatePath(...)`.

## База данных и деньги

- Схему меняем только через `prisma/schema.prisma` + `npx prisma migrate dev --name <слаг>`.
- Применённые миграции не редактируются — только новая миграция.
- Деструктивная миграция (удаление таблицы/колонки с данными) — явно предупреди
  пользователя до применения и отметь это в описании изменений.
- Даты в БД — `@db.Timestamptz(3)`. В интерфейсе даты показываем через
  `formatDate` из `lib/domain/dates.ts` (Москва).
- **Деньги — всегда целые копейки (`amountMinor: Int`)**. Форматирование —
  только `formatMoney`, разбор пользовательского ввода — только `parseMoneyToMinor`
  (оба из `lib/domain/money.ts`). Никаких float для денег.
  Исключение: суммы, способные превысить лимит Int (±21,4 млн ₽), — `BigInt`
  в схеме и `formatMoneyBig` для показа (модуль заявок на оплату). BigInt
  нельзя передавать в клиентские компоненты — форматируй в строку на сервере.

## Тесты (гибрид по слоям)

- Чистая логика в `lib/domain/` — обязательные unit-тесты (Vitest, TDD),
  файл `*.test.ts` рядом с кодом.
- Ключевые пользовательские сценарии — e2e-смоук в `tests/e2e/` (Playwright).
  При добавлении фичи добавь/обнови смоук её главного сценария. E2e-тесты
  не должны зависеть от seed-данных — создавай данные внутри теста.
- **Unit-тесты на React-компоненты и страницы запрещены** — их роль выполняет e2e.

## UI: только shadcn в рамках темы

1. Компоненты — только shadcn/ui: `npx shadcn@latest add <имя>` или композиции из уже
   установленных примитивов. Другие UI-библиотеки запрещены.
2. Стили — только Tailwind-классы и токены темы (CSS-переменные из `app/globals.css`).
   Без хардкод-цветов, инлайн-стилей и новых шрифтов. Тему меняет только разработчик.
3. Иконки — только `lucide-react`.
4. Графики — только shadcn charts.
5. Нет подходящего компонента в shadcn — остановись и спроси разработчика.
   Одобренные исключения фиксируются в списке ниже.

### Источник правды

- В проекте установлен **официальный скилл shadcn** (`.claude/skills/shadcn/`,
  ставится через `npx skills add shadcn/ui`, версии фиксирует `skills-lock.json`).
  Он загружается автоматически при работе с UI и сам подтягивает конфигурацию
  проекта (`npx shadcn@latest info --json`), документацию и правила композиции —
  следуй его правилам (`rules/`) при написании компонентов.
- Перед созданием или правкой любого UI-компонента сверяйся с реестром shadcn:
  через инструменты MCP-сервера `shadcn` (если подключён) или командами CLI:
  - `npx shadcn@latest docs <component>` — дока, API-референс и примеры компонента;
  - `npx shadcn@latest search <registry> -q <query>` — поиск по реестру
    (например, `search @shadcn -q card`);
  - `npx shadcn@latest view <item>` — просмотр элемента реестра
    (например, `view @shadcn/card`).
- Не пиши компоненты shadcn по памяти: примеры из training data часто
  устаревшие. Актуальная документация всегда важнее памяти модели.
- Индекс всей документации для ИИ: https://ui.shadcn.com/llms.txt
- MCP-сервер `shadcn` подключён (`.mcp.json`); статус — через `/mcp`.
  Если MCP недоступен, CLI-команды выше работают в любом случае.

### Приоритет реестра

- Если компонент есть в реестре — установи его
  (`npx shadcn@latest add <name>`) и используй.
- Запрещено писать самописные аналоги того, что уже есть в реестре
  (кнопки, диалоги, формы, таблицы и т.д.).

### Стилизация

- Только Tailwind-классы. Inline-стили и CSS-модули запрещены.
- Цвета — только через семантические токены темы из `app/globals.css`
  (`bg-primary`, `text-muted-foreground`, `border-border`, `--chart-1..5`).
- Запрещены raw-палитра (`text-red-500`, `bg-neutral-100`), `#hex` и
  литеральные цветовые функции (`oklch(…)`, `rgb(…)`) в компонентах.
  Ручные `dark:`-переопределения цветов не нужны — семантические токены сами
  темизируются; `dark:` допустим только там, где так пишет сам shadcn.

### Слои

- `components/ui/` — стоковые атомы shadcn. Не редактировать; обновление
  только через `npx shadcn@latest add`.
- Авторские композиции — в `components/` (вне `ui/`). Данные попадают в них
  только через пропсы: композиции не импортируют фикстуры.

### Процесс UI-работы

- Сложные страницы: сначала план — структура страницы + список компонентов
  реестра, потом код (в finflow это часть обычного цикла brainstorming).
- Собирай страницы из блоков-секций (hero, stats, таблица — по одной за раз),
  а не одним полотном: качество каждого модуля контролируется отдельно.
- Фронтенд наращивай вертикальными срезами (tracer bullet): один роут → его
  layout и тема → один работающий экран → следующий срез. Не строй
  «все компоненты, потом все страницы».
- После каждого изменения: `npm run typecheck` + `npm run lint`, затем проверка
  в браузере — обе темы (если в проекте есть тёмная), мобильная ширина,
  чистая консоль. Часть ошибок (например, семантика кнопки-ссылки в Base UI)
  видна только в живом браузере — ни typecheck, ни SSR их не ловят.
- Document & Clear: если контекст диалога разросся и агент начинает ошибаться —
  зафиксируй текущий план/состояние в файл, сбрось сессию (`/clear`) и начни
  новую с чтения этого файла.

### Специфика Base UI

- Примитивы проекта — **Base UI** (`@base-ui/react`, стиль `base-mira`),
  НЕ Radix. Примеры из training data почти всегда Radix-эпохи — не копируй
  их дословно.
- Это касается и живых примеров из реестра (MCP `get_item_examples` / `view`):
  они могут быть в стиле new-york-v4 с `asChild` — структуру бери оттуда,
  но API адаптируй под Base UI (правило `rules/base-vs-radix.md` скилла).
- `asChild` в Base UI не существует. Чтобы отрендерить компонент другим
  элементом, используй проп `render`: `<Button render={<a href="…" />}>`.
- `Button` в роли ссылки/анкора требует `nativeButton={false}` — иначе
  Base UI бросает runtime-ошибку о потерянной семантике кнопки, и видна
  она только в консоли живого браузера (typecheck/lint/SSR её не ловят).

## Языковые конвенции

- Интерфейс приложения — на русском.
- Код, идентификаторы, имена файлов — на английском.
- Коммиты — conventional commits (`feat:`, `fix:`, `chore:`, `docs:`), описание на русском.

## Проверки перед любым коммитом

```bash
npm run format && npm run lint && npm run typecheck && npm run test
```
